# 工作方法论

跨项目通用的编码行为准则。每次对话自动加载。

## 1. 先想再写

遇到需求有多种理解时，先列出可能的方案和各自的 tradeoff，让用户选择，不要默默选一个实现。

- 不确定的地方明确说出来，不要猜
- 有更简单的方案就提出来，即使用户没问
- 搞不清楚就停下来问，不要带着困惑往下写

## 2. 最小必要

只写解决当前问题所需的代码，不多不少。

- 不加没要求的功能、抽象、"灵活性"
- 单次使用的逻辑不要提前抽象
- 200 行能用 50 行写就重写
- 自检：一个资深工程师会说这过度设计了吗？如果会，简化

## 3. 精准改动

每一行改动都必须能直接追溯到用户的请求。

- 不"顺手改进"旁边的代码、注释、格式
- 匹配现有代码风格，即使你觉得不好
- 你的改动产生的孤儿代码（import、变量）要清理，别人的不要动
- 发现无关问题可以提一句，但不要动手改

## 4. 目标驱动

把任务转化为可验证的目标，而不是模糊的步骤。

- "修 bug" → 先写复现测试，再修，确认测试通过
- "加功能" → 先定义成功标准，再实现，逐步验证
- 多步任务列计划，每步有验证方式：
  ```
  1. [步骤] → 验证：[怎么确认成功]
  2. [步骤] → 验证：[怎么确认成功]
  ```

## 5. 先读再写

改代码前先读相关代码，理解现有模式再动手。

- 复用现有模块、类型、工具，不重复造轮子
- 不臆造项目中不存在的字段、状态、API
- 不确定某个函数/模块是否存在时，先搜索确认
