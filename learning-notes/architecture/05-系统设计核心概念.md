### 一致性哈希

```
传统哈希的问题：
  hash(key) % N → 节点增减时，几乎所有 key 都要重新映射

一致性哈希：
  将节点和 key 都映射到一个环上，key 顺时针找到最近的节点

       Node A
        ╱  ╲
      ╱      ╲
  Node D      Node B
      ╲      ╱
        ╲  ╱
       Node C

  增加 Node E：只影响 E 和相邻节点之间的 key
  删除 Node B：只影响 B 的 key 迁移到下一个节点

  虚拟节点：每个物理节点映射多个虚拟节点，解决数据倾斜
```

## 3. 缓存

```
┌─────────── 缓存策略 ───────────┐
│                                 │
│  Cache-Aside（旁路缓存）⭐ 最常用│
│  读：先查缓存 → 未命中 → 查 DB → 写缓存
│  写：更新 DB → 删除缓存          │
│                                 │
│  Read-Through（读穿透）          │
│  缓存层自动从 DB 加载            │
│                                 │
│  Write-Through（写穿透）         │
│  写缓存时同步写 DB               │
│                                 │
│  Write-Behind（写回）            │
│  写缓存后异步批量写 DB           │
│  性能最好，但有数据丢失风险       │
└─────────────────────────────────┘
```

### 缓存常见问题

```python
# 缓存穿透：查询不存在的数据，每次都打到 DB
# 解决：布隆过滤器 + 缓存空值
def get_user(user_id: str):
    # 布隆过滤器快速判断是否存在
    if not bloom_filter.might_contain(user_id):
        return None

    cached = redis.get(f"user:{user_id}")
    if cached is not None:
        return json.loads(cached) if cached != "NULL" else None

    user = db.find_user(user_id)
    if user:
        redis.setex(f"user:{user_id}", 3600, json.dumps(user))
    else:
        redis.setex(f"user:{user_id}", 300, "NULL")  # 缓存空值，短过期
    return user


# 缓存雪崩：大量 key 同时过期，请求全打到 DB
# 解决：过期时间加随机偏移
import random

def cache_with_jitter(key: str, value: str, base_ttl: int = 3600):
    jitter = random.randint(0, 600)  # 0~10 分钟随机偏移
    redis.setex(key, base_ttl + jitter, value)


# 缓存击穿：热点 key 过期瞬间，大量请求打到 DB
# 解决：互斥锁（只允许一个请求回源）
def get_hot_data(key: str):
    cached = redis.get(key)
    if cached:
        return json.loads(cached)

    # 互斥锁：只有一个请求去查 DB
    lock_key = f"lock:{key}"
    if redis.set(lock_key, "1", nx=True, ex=10):
        try:
            data = db.query(key)
            redis.setex(key, 3600, json.dumps(data))
            return data
        finally:
            redis.delete(lock_key)
    else:
        time.sleep(0.1)
        return get_hot_data(key)  # 等待后重试
```

## 4. 限流

```
┌─────────── 限流算法 ───────────┐
│                                 │
│  1. 固定窗口                     │
│     每分钟最多 100 次            │
│     问题：窗口边界突发            │
│                                 │
│  2. 滑动窗口                     │
│     统计过去 60 秒的请求数       │
│     更平滑，但内存开销大          │
│                                 │
│  3. 令牌桶（Token Bucket）⭐     │
│     固定速率放入令牌              │
│     请求消耗令牌，桶空则拒绝      │
│     允许突发流量（桶内有存量）     │
│                                 │
│  4. 漏桶（Leaky Bucket）        │
│     请求进入桶，固定速率流出      │
│     严格平滑，不允许突发          │
└─────────────────────────────────┘
```

```python
import time

class TokenBucket:
    """令牌桶限流器"""

    def __init__(self, rate: float, capacity: int):
        self.rate = rate            # 每秒放入令牌数
        self.capacity = capacity    # 桶容量
        self.tokens = capacity      # 当前令牌数
        self.last_refill = time.time()

    def allow(self) -> bool:
        """是否允许请求通过"""
        now = time.time()
        # 补充令牌
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last_refill = now

        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

# 每秒 100 个请求，允许突发 200
limiter = TokenBucket(rate=100, capacity=200)

if limiter.allow():
    handle_request()
else:
    return_429_too_many_requests()
```

## 6. CAP 定理

```
┌─────────── CAP 定理 ───────────┐
│                                 │
│  分布式系统只能同时满足两个：     │
│                                 │
│         C（一致性）              │
│          ╱ ╲                    │
│         ╱   ╲                   │
│        ╱ CA  ╲                  │
│       ╱(单机DB)╲                │
│      ╱─────────╲               │
│     A           P               │
│  (可用性)    (分区容错)          │
│                                 │
│  网络分区一定会发生，所以实际选择：│
│  ├─ CP：保一致性，牺牲可用性     │
│  │   ZooKeeper, etcd, HBase    │
│  └─ AP：保可用性，牺牲一致性     │
│      Cassandra, DynamoDB, Eureka│
│                                 │
│  实际中更常用 BASE：              │
│  Basically Available            │
│  Soft state                     │
│  Eventually consistent          │
│  （基本可用 + 最终一致性）        │
└─────────────────────────────────┘
```

## 7. 常见系统设计模板

```
系统设计面试四步法：
│
├─ Step 1: 需求澄清（5 分钟）
│   ├─ 功能需求：核心功能是什么？
│   ├─ 非功能需求：QPS、延迟、可用性
│   └─ 约束：数据量、用户量、预算
│
├─ Step 2: 高层设计（10 分钟）
│   ├─ 画出核心组件和数据流
│   ├─ 确定 API 设计
│   └─ 确定数据模型
│
├─ Step 3: 详细设计（20 分钟）
│   ├─ 深入核心组件
│   ├─ 数据库选型和 Schema
│   ├─ 缓存策略
│   └─ 消息队列设计
│
└─ Step 4: 扩展与优化（10 分钟）
    ├─ 瓶颈分析
    ├─ 扩展方案（水平/垂直）
    ├─ 容灾和高可用
    └─ 监控和告警
```

## 8. 经典系统设计速查

| 系统 | 核心挑战 | 关键技术 |
|------|---------|---------|
| 短链接服务 | 高并发读、ID 生成 | Base62 编码、布隆过滤器、缓存 |
| 消息系统 | 实时性、消息顺序 | WebSocket、消息队列、分区有序 |
| 新闻 Feed | 读扩散 vs 写扩散 | Fan-out、时间线缓存、推拉结合 |
| 搜索引擎 | 倒排索引、相关性 | Elasticsearch、TF-IDF、分片 |
| 限流系统 | 分布式限流、精度 | Redis + Lua、令牌桶、滑动窗口 |
| 秒杀系统 | 瞬时高并发、超卖 | 预扣库存、队列削峰、乐观锁 |
| 分布式 ID | 全局唯一、有序 | Snowflake、Leaf、UUID v7 |
