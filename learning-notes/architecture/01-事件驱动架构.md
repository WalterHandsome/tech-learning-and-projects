# 事件驱动架构
‍‍​​​​​​​​​‌​‌​‌‌‌​​​​​​​​​‌‌​​​​‌​​​​​​​​​‌‌​‌‌​​​​​​​​​​​‌‌‌​‌​​​​​​​​​​​‌‌​​‌​‌​​​​​​​​​‌‌‌​​‌​​​​​​​​​​​‌​​​​​​​​​​​​​​‌​‌​‌‌‌​​​​​​​​​‌‌​​​​‌​​​​​​​​​‌‌​‌‌‌​​​​​​​​​​‌‌​​‌‌‌‍‍
> Author: Walter Wang

## 1. 概述

事件驱动架构（Event-Driven Architecture, EDA）通过事件实现服务间的解耦和异步通信。核心思想：服务之间不直接调用，而是通过发布/订阅事件来协作。

```
┌──────────── 事件驱动 vs 请求驱动 ────────────┐
│                                                │
│  请求驱动（同步）：                             │
│  订单服务 ──HTTP──→ 库存服务 ──HTTP──→ 支付服务 │
│  （强耦合，链式调用，一个挂全挂）                │
│                                                │
│  事件驱动（异步）：                             │
│  订单服务 ──→ [订单已创建] ──→ Event Broker     │
│                                  │    │    │   │
│                              库存服务 支付 通知  │
│  （松耦合，各自独立消费，互不影响）              │
└────────────────────────────────────────────────┘
```

## 2. 核心概念

```
事件驱动四要素：
├─ Event（事件）：已发生的不可变事实，如"订单已创建"
├─ Producer（生产者）：发布事件，不关心谁消费
├─ Consumer（消费者）：订阅事件，做出响应
└─ Broker（消息中间件）：事件的传输和存储
```

### 事件的三种类型

```
┌─────────── 事件类型 ───────────┐
│                                 │
│  1. 通知事件（Notification）     │
│     只告诉你"发生了什么"         │
│     { "type": "order.created",  │
│       "orderId": "123" }        │
│     消费者需要回查获取详情        │
│                                 │
│  2. 状态转移事件（State Transfer）│
│     携带完整数据                  │
│     { "type": "order.created",  │
│       "order": { 全部字段 } }    │
│     消费者无需回查                │
│                                 │
│  3. 领域事件（Domain Event）     │
│     DDD 中的业务事件             │
│     { "type": "OrderPlaced",    │
│       "aggregateId": "123",     │
│       "items": [...] }          │
│     绑定聚合根，有业务语义        │
└─────────────────────────────────┘
```

## 3. 两大架构模式

### 3.1 发布-订阅模式（Pub/Sub）

```
Producer ──→ Topic ──→ Consumer A
                  ├──→ Consumer B
                  └──→ Consumer C

特点：
├─ 一对多广播
├─ 消费者独立消费，互不影响
├─ 适合通知、日志、监控
└─ 消息消费后可能丢失（取决于实现）
```

```python
# Kafka 发布-订阅示例
from confluent_kafka import Producer, Consumer

# --- 生产者 ---
producer = Producer({"bootstrap.servers": "localhost:9092"})

def publish_order_event(order: dict):
    """发布订单事件"""
    event = {
        "type": "order.created",
        "timestamp": datetime.utcnow().isoformat(),
        "data": order,
    }
    producer.produce(
        topic="order-events",
        key=order["order_id"],      # 同一订单的事件发到同一分区，保证顺序
        value=json.dumps(event),
    )
    producer.flush()

# --- 消费者 ---
consumer = Consumer({
    "bootstrap.servers": "localhost:9092",
    "group.id": "inventory-service",    # 消费者组
    "auto.offset.reset": "earliest",
})
consumer.subscribe(["order-events"])

while True:
    msg = consumer.poll(1.0)
    if msg is None:
        continue
    event = json.loads(msg.value())
    if event["type"] == "order.created":
        deduct_inventory(event["data"]["items"])
```

### 3.2 事件流模式（Event Streaming）

```
Event Stream（不可变日志）：
┌───┬───┬───┬───┬───┬───┬───┐
│ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ ← offset
└───┴───┴───┴───┴───┴───┴───┘
  ↑               ↑           ↑
Consumer A    Consumer B    最新写入

特点：
├─ 事件持久化，可重放
├─ 多个消费者各自维护消费位置（offset）
├─ 适合事件溯源、数据管道、实时分析
└─ Kafka 是典型实现
```

## 4. 消息中间件对比

| 维度 | Kafka | RabbitMQ | Redis Streams | Pulsar |
|------|-------|----------|---------------|--------|
| 模型 | 分布式日志 | 消息队列 | 内存流 | 分布式日志 |
| 吞吐量 | 百万/秒 | 万级/秒 | 十万/秒 | 百万/秒 |
| 消息持久化 | 磁盘，可配置保留 | 可选持久化 | 内存+AOF | 分层存储 |
| 消息顺序 | 分区内有序 | 队列内有序 | Stream 内有序 | 分区内有序 |
| 消费模式 | Pull | Push/Pull | Pull | Push/Pull |
| 消息回溯 | ✅ 按 offset 回放 | ❌ 消费即删 | ✅ 按 ID 回放 | ✅ 按 offset 回放 |
| 延迟 | 毫秒级 | 微秒级 | 微秒级 | 毫秒级 |
| 适用场景 | 大数据管道、事件溯源 | 任务队列、RPC | 轻量实时、缓存旁路 | 多租户、跨地域 |
| 运维复杂度 | 高（ZK/KRaft） | 中 | 低 | 高 |

```
选型建议：
├─ 高吞吐 + 事件回溯 → Kafka
├─ 低延迟 + 灵活路由 → RabbitMQ
├─ 已有 Redis + 轻量需求 → Redis Streams
├─ 多租户 + 跨地域 → Pulsar
└─ 不确定？ → Kafka（生态最大，社区最活跃）
```

## 5. 关键设计模式

### 5.1 幂等消费

消息可能重复投递，消费者必须保证幂等。

```python
class IdempotentConsumer:
    """幂等消费者：同一事件只处理一次"""

    def __init__(self, redis_client):
        self.redis = redis_client

    def process(self, event: dict):
        event_id = event["event_id"]

        # 检查是否已处理（用 Redis SET NX 原子操作）
        if not self.redis.set(f"processed:{event_id}", "1", nx=True, ex=86400):
            return  # 已处理过，跳过

        try:
            self._handle(event)
        except Exception:
            self.redis.delete(f"processed:{event_id}")  # 处理失败，允许重试
            raise

    def _handle(self, event: dict):
        """实际业务处理"""
        if event["type"] == "order.created":
            deduct_inventory(event["data"])
```

### 5.2 死信队列（DLQ）

处理失败的消息进入死信队列，避免阻塞正常消费。

```
正常流程：
Topic ──→ Consumer ──→ 处理成功 ✅

失败流程：
Topic ──→ Consumer ──→ 重试 3 次 ──→ 仍失败 ──→ Dead Letter Queue
                                                      │
                                                  人工排查 / 告警
```

### 5.3 Saga 模式（分布式事务）

跨服务的事务通过事件编排，每步失败触发补偿操作。

```
┌─────────── Saga：订单流程 ───────────┐
│                                       │
│  正常流程：                            │
│  创建订单 → 扣减库存 → 扣款 → 完成    │
│                                       │
│  补偿流程（扣款失败）：                 │
│  创建订单 → 扣减库存 → 扣款失败 ✗      │
│                 ↓                     │
│            恢复库存 → 取消订单          │
│                                       │
│  两种实现：                            │
│  ├─ 编排式（Choreography）             │
│  │   服务间通过事件自协调               │
│  │   简单，但流程分散难追踪             │
│  └─ 协调式（Orchestration）            │
│      中心协调器控制流程                 │
│      清晰，但协调器是单点               │
└───────────────────────────────────────┘
```

```python
# 协调式 Saga 示例
class OrderSaga:
    """订单 Saga 协调器"""

    def __init__(self, event_bus):
        self.event_bus = event_bus
        self.steps = [
            SagaStep("create_order", self.create_order, self.cancel_order),
            SagaStep("reserve_stock", self.reserve_stock, self.release_stock),
            SagaStep("process_payment", self.process_payment, self.refund_payment),
        ]

    async def execute(self, order_data: dict):
        completed = []
        for step in self.steps:
            try:
                await step.action(order_data)
                completed.append(step)
            except Exception as e:
                # 失败：逆序执行补偿
                for s in reversed(completed):
                    await s.compensate(order_data)
                raise SagaFailed(f"{step.name} 失败: {e}")

        await self.event_bus.publish("order.completed", order_data)
```

### 5.4 Outbox 模式

解决"数据库写入 + 事件发布"的原子性问题。

```
问题：
  1. 写数据库 ✅
  2. 发事件 ✗（网络故障）
  → 数据不一致

Outbox 解决方案：
  1. 在同一个数据库事务中：
     ├─ 写业务表
     └─ 写 outbox 表（待发送事件）
  2. 独立进程轮询 outbox 表，发送事件到 Broker
  3. 发送成功后标记已处理

┌─────────────────────────────────┐
│  数据库事务（原子）               │
│  ├─ INSERT INTO orders (...)    │
│  └─ INSERT INTO outbox (...)    │
└──────────────┬──────────────────┘
               │
    ┌──────────┴──────────┐
    │ Outbox Poller（轮询）│
    └──────────┬──────────┘
               │
         Event Broker
```

## 7. 常见陷阱

```
事件驱动的坑：
├─ 事件顺序：跨分区无法保证全局顺序
│   → 用 partition key 保证同一实体的事件有序
├─ 消息丢失：生产者发送失败 / Broker 宕机
│   → 开启 acks=all + 副本同步
├─ 消息重复：网络抖动导致重复投递
│   → 消费者必须幂等
├─ 事件风暴：一个事件触发大量下游事件
│   → 限流 + 背压（backpressure）
├─ 调试困难：异步链路难以追踪
│   → 全链路 Trace ID + 集中日志
└─ 最终一致性：数据不是立即一致的
    → 业务上接受延迟，UI 上给用户预期
```
