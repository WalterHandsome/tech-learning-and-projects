## 1. MVVM vs MVI

```kotlin
// MVVM: ViewModel 暴露多个 State/LiveData，View 分别观察
class MvvmViewModel : ViewModel() {
    val users = MutableStateFlow<List<User>>(emptyList())
    val isLoading = MutableStateFlow(false)
    val error = MutableStateFlow<String?>(null)
    // 问题：多个状态可能不一致
}

// MVI: 单一 State + 单向数据流
class MviViewModel : ViewModel() {
    private val _state = MutableStateFlow(UiState())
    val state: StateFlow<UiState> = _state.asStateFlow()

    fun onIntent(intent: UserIntent) {
        // Intent → Reducer → State → UI
        // 优点：状态一致性，可预测，易测试
        // 缺点：模板代码多，简单场景过度设计
    }
}

data class UiState(
    val users: List<User> = emptyList(),
    val isLoading: Boolean = false,
    val error: String? = null
)

// 选择建议：
// 简单页面 → MVVM
// 复杂交互/状态多 → MVI
// 团队统一 → 选一种坚持
```

## 2. Hilt vs Koin

```kotlin
// Hilt（编译时）
// ✅ 编译时检查依赖图，错误提前发现
// ✅ Google 官方支持，与 Jetpack 深度集成
// ❌ 编译时间增加，注解处理器
// ❌ 学习曲线较高

@HiltViewModel
class UserViewModel @Inject constructor(
    private val repository: UserRepository
) : ViewModel()

// Koin（运行时）
// ✅ DSL 简洁，上手快
// ✅ 纯 Kotlin，无注解处理
// ❌ 运行时解析，错误延迟到运行时
// ❌ 大型项目性能略差

val viewModel: UserViewModel by viewModel()

// 选择建议：
// 大型项目/团队 → Hilt（编译时安全）
// 小型项目/快速原型 → Koin（简洁高效）
// KMM 项目 → Koin（跨平台支持）
```

## 3. 内存泄漏检测与解决

```kotlin
// 常见泄漏场景及解决方案

// 1. 匿名内部类持有外部引用
// ❌ Handler/Runnable 持有 Activity
// ✅ 使用 lifecycleScope 或 WeakReference

// 2. 单例持有 Activity Context
// ❌ Singleton.init(activity)
// ✅ Singleton.init(activity.applicationContext)

// 3. 未注销监听器
// ❌ 注册了 BroadcastReceiver 但未 unregister
// ✅ 在 onStop/onDestroy 中注销

// 4. Coroutine 泄漏
// ❌ GlobalScope.launch { /* 持有 Activity 引用 */ }
// ✅ viewModelScope.launch / lifecycleScope.launch

// 5. Fragment ViewBinding 泄漏
// ❌ private val binding = FragmentBinding.inflate(...)
// ✅ onDestroyView 中 _binding = null

// 检测工具
// LeakCanary: debug 自动检测
// Android Profiler: Memory → Dump Heap
// StrictMode: 检测主线程 IO
```

## 4. Compose 重组优化

```kotlin
// 重组触发条件：State 变化时，读取该 State 的 Composable 重组

// 1. 稳定性标注
@Immutable  // 所有属性不可变
data class UserUiModel(val id: String, val name: String)

@Stable  // 可变但变化可被 Compose 追踪
class CounterState {
    var count by mutableStateOf(0)
}

// 2. 避免不稳定参数
// ❌ 每次重组创建新 List
@Composable
fun UserList() {
    val users = listOf(User("1", "张三"))  // 每次新对象
    ItemList(users)
}

// ✅ 使用 remember
@Composable
fun UserList() {
    val users = remember { listOf(User("1", "张三")) }
    ItemList(users)
}

// 3. key 优化 LazyColumn
LazyColumn {
    items(users, key = { it.id }) { user ->
        UserItem(user)  // key 帮助识别项目，减少重组
    }
}

// 4. derivedStateOf 减少重组频率
val showButton by remember {
    derivedStateOf { listState.firstVisibleItemIndex > 5 }
}

// 5. 检查重组次数
// Layout Inspector → Compose → Recomposition Counts
// Compose Compiler Metrics 分析稳定性
```

## 5. 启动优化策略

```kotlin
// 冷启动优化目标：< 500ms

// 1. Application.onCreate 优化
// - 必要初始化放主线程
// - 非必要初始化异步执行
// - 使用 App Startup 管理初始化顺序

// 2. 首屏优化
// - SplashScreen API 替代自定义启动页
// - 延迟加载非首屏内容
// - 使用 ViewStub / LazyColumn

// 3. Baseline Profiles
// - 预编译热路径代码
// - 首次启动速度提升 30%+

// 4. 测量工具
// adb shell am start-activity -W com.example/.MainActivity
// Macrobenchmark 自动化测量
@RunWith(AndroidJUnit4::class)
class StartupBenchmark {
    @get:Rule
    val rule = MacrobenchmarkRule()

    @Test
    fun startupCompilation() = rule.measureRepeated(
        packageName = "com.example.app",
        metrics = listOf(StartupTimingMetric()),
        iterations = 5,
        startupMode = StartupMode.COLD
    ) {
        pressHome()
        startActivityAndWait()
    }
}
```
