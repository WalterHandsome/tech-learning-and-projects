# 支付行业术语小白指南

> 本文档面向支付行业零基础读者，结合 AgentToken 笔记中出现的所有术语，用通俗语言逐一解释。建议配合 01-支付与Agent支付知识深度详解 一起阅读，本文档可作为术语字典随时查阅。

## 一、银行卡与卡号相关

### PAN（Primary Account Number，主账号）

就是你信用卡/借记卡正面那串 16 位数字（如 4242 4242 4242 4242）。它是银行卡的"身份证号"，所有交易都围绕它进行。

为什么重要：PAN 一旦泄露，别人就能盗刷你的卡。所以支付行业花了大量精力来"藏住"这个号码，这就引出了下面的 Tokenization。

### CVV / CVC（Card Verification Value）

卡背面的 3 位安全码（Amex 是 4 位）。线上支付时用来证明"你手里确实有这张卡"。

### 卡组织 / Card Scheme / Card Network

Visa、Mastercard、银联、American Express 这些就是卡组织。它们不直接发卡，而是运营一个全球支付网络：

- 制定交易规则和标准
- 在发卡行和收单行之间路由交易
- 提供欺诈检测、争议处理等基础设施

类比：卡组织就像高速公路运营商，银行是收费站，你的钱在高速上跑。

### 发卡行 / Issuer / Issuing Bank

给你发信用卡的银行。比如招商银行给你发了一张 Visa 卡，招行就是发卡行。

职责：审批交易、从你账户扣款、承担信用风险。

### 收单行 / Acquirer / Acquiring Bank

为商户提供收款服务的银行或机构。商户的 POS 机、线上支付接口背后都有收单行。

职责：接收交易请求、与卡组织通信、把钱结算给商户。

### MCC（Merchant Category Code，商户类别码）

每个商户都有一个 4 位数的分类代码，标识它的行业类型。比如：
- 5411 = 超市/杂货店
- 4511 = 航空公司
- 5812 = 餐饮

AgentToken 用 MCC 来做策略控制：比如限制某个 Agent 只能在航空类商户消费。


## 二、Tokenization（令牌化）

### 什么是 Tokenization？

把敏感的卡号（PAN）替换成一串没有实际价值的随机字符（Token）。即使 Token 被盗，攻击者也无法用它直接消费。

生活类比：你去酒店寄存行李，前台给你一个取件牌。取件牌本身不值钱，但凭它可以取到你的行李。Token 就是这个"取件牌"，PAN 就是你的"行李"。

### Gateway Token（网关令牌）vs Network Token（网络令牌）

这是两种不同层级的 Tokenization：

| 对比项 | Gateway Token | Network Token |
|--------|--------------|---------------|
| 谁生成的 | 支付网关（如 Stripe、Adyen） | 卡组织（Visa、Mastercard） |
| 作用范围 | 只在该网关内有效 | 全网络通用 |
| 安全性 | 中等 | 更高（附带一次性密码学凭证） |
| 卡过期处理 | 需要手动更新 | 卡组织自动同步新卡信息 |
| 交易成功率 | 一般 | 更高（发卡行更信任网络令牌） |

AgentToken 笔记中的 "Network Token" 就是卡组织级别的令牌，安全性最高。

### Cryptogram（密码学凭证）

Network Token 在每次交易时会附带一个一次性的 Cryptogram，类似"动态验证码"。即使 Token 被截获，没有对应的 Cryptogram 也无法完成交易。

## 三、虚拟卡相关

### VCN（Virtual Card Number，虚拟卡号）

不是实体卡，而是系统生成的一组卡号 + 有效期 + CVV。可以像真卡一样在线上消费，但通常有更严格的限制。

### 一次性虚拟卡（Single-use VCN）

用完即废的虚拟卡号。特点：
- 只能用一次（或限定金额/商户/时间）
- 用完自动失效
- 即使泄露也无法重复使用

AgentToken 中的 "One-time VCN" 就是这个：给 AI Agent 发一张只能用一次的虚拟卡，买完东西卡就作废了。

### 预付卡 / Prepaid Card

先充值、后消费的卡。余额用完就不能再刷。AgentToken 的虚拟卡本质上就是预付模式——先设定消费上限（Spend Limit），Agent 在额度内消费。

## 四、合规与身份验证

### KYC（Know Your Customer，了解你的客户）

金融机构在开户前必须验证客户身份的流程。通常包括：
- 收集身份证件（护照、身份证）
- 验证地址证明
- 对照制裁名单和反洗钱数据库筛查

目的：防止身份欺诈、洗钱、恐怖融资等违法行为。

日常例子：你去银行开户时要出示身份证、填表、拍照——这就是 KYC。

### KYB（Know Your Business，了解你的企业）

KYC 的企业版。验证的不是个人，而是公司：
- 公司注册信息、营业执照
- 股权结构、实际控制人（UBO）
- 财务状况、经营合规性

AgentToken 中个人用户走 KYC 流程，企业用户走 KYB 流程。

### AML（Anti-Money Laundering，反洗钱）

一套法律法规和操作流程，用来检测和防止通过金融系统洗钱。KYC/KYB 是 AML 的一部分。

### PCI DSS（Payment Card Industry Data Security Standard）

支付卡行业数据安全标准。任何存储、处理或传输卡号数据的机构都必须遵守。要求包括：
- 加密存储卡数据
- 限制访问权限
- 定期安全审计

AgentToken 用 Vault/HSM 存储敏感数据，就是为了满足 PCI DSS 要求。

## 五、安全存储

### Vault（保险库）

专门用来安全存储敏感数据（如卡号、密钥）的系统。数据加密存储，只有授权的服务才能访问。

### HSM（Hardware Security Module，硬件安全模块）

一种专用硬件设备，用来生成、存储和管理加密密钥。比软件加密更安全，因为密钥永远不会离开硬件。

类比：Vault 是一个上了锁的保险柜，HSM 是保险柜里那把永远拿不出来的钥匙。

## 六、交易流程相关

### PaymentIntent（支付意图）

现代支付 API（如 Stripe）中的核心概念。代表"一笔即将发生的支付"，包含金额、货币、支付方式等信息。

在 AgentToken 中，创建 Token 时会对用户的支付方式发起一个 PaymentIntent hold（预授权），冻结对应金额，确保 Agent 消费时有足够资金。

### 预授权 / Authorization Hold

先冻结一笔金额，但不实际扣款。等交易确认后再真正扣款（Capture）。

例子：酒店入住时刷卡预授权 2000 元，退房时按实际消费扣款。AgentToken 的 Spend Limit 就是通过预授权实现的。

### Interchange Fee（交换费）

每笔卡交易中，收单行付给发卡行的手续费，通常是交易金额的 1.5%–3.5%。这是卡组织支付网络的"过路费"。

这也是为什么虚拟卡不适合微支付（比如 $0.005 的搜索查询）——手续费比交易本身还贵。

## 七、AI Agent 支付协议

### X402 协议

由 Coinbase 推出的开源协议，复活了 HTTP 规范中沉睡多年的 402 "Payment Required" 状态码。工作方式：

1. Agent 请求一个 API → 服务器返回 HTTP 402 + 价格信息
2. Agent 用稳定币（如 USDC）签名支付
3. 服务器验证支付 → 返回数据

特点：无需注册、无需 API Key、无需信用卡，直接用加密货币按次付费。适合 Agent 之间的微支付。

来源：[x402 protocol architecture](https://chainstack.com/x402-protocol-for-ai-agents/)

### Stablecoin（稳定币）

价值锚定法定货币（通常是美元）的加密货币。如 USDC（1 USDC ≈ 1 美元）。

为什么 Agent 支付用稳定币：波动小、可编程、结算快、无需银行账户。

### Verifiable Credential / VC（可验证凭证）

一种数字证书标准（W3C 制定），可以证明某个实体的某些属性，且可以被第三方验证真伪。

在 AgentToken 中，X402 VC 是一种支付凭证：它不仅包含支付信息，还能附带"谁授权的、用途是什么、有什么限制"等可验证属性。

### ACP（Agentic Commerce Protocol）

由 Stripe 和 OpenAI 联合推出的协议，专注于 Agent 在商户网站完成结账的流程。已在 ChatGPT 中上线使用。

### UCP（Universal Commerce Protocol）

由 Google 和 Shopify 推出，覆盖从商品发现到履约的完整购物旅程，范围比 ACP 更广。

### AP2（Agent Payments Protocol）

由 Google 推出，专注于 Agent 之间的授权和信任链，使用去中心化标识符和可验证凭证。

### MCP（Model Context Protocol）

不是支付协议，而是 AI 模型与外部工具交互的协议。AgentToken 提到需要一个"协议友好的中间层"，就是因为 Agent 可能通过 MCP 等协议来调用支付能力。

## 八、AgentToken 特有概念

### User（用户）

AgentToken 系统的顶层实体，分两种：
- Individual User（个人用户）：个人，走 KYC
- Corporate User（企业用户）：公司/组织，走 KYB

User 可以管理 API Key，但不能直接发起交易。

### Member（成员）

User 下面的真人角色，必须绑定支付方式才能申请 Token。

关键设计：Token 的持有者必须是人（不是 Agent），这是合规要求——确保每笔支付背后都有一个可追责的真人。

### Token（令牌）

Agent 实际用来支付的凭证。由 Member 申请，有消费上限、有效期、商户限制等策略约束。

层级关系：User → Member（绑定支付方式）→ Token（给 Agent 用）

### Spend Limit（消费上限）

Token 的最大可消费金额。通过预授权冻结在用户的支付方式上，是一个硬性限制——Agent 无法超额消费。

### Policy（策略）

控制 Token 使用范围的规则集合，可以包括：
- 金额上限
- 时间窗口（如 24 小时内有效）
- 商户/MCC 限制（如只能在航空类商户使用）
- 地理限制
- 一次性 vs 可重复使用
- 频率限制

## 九、开发者相关

### API Key

访问 AgentToken API 的密钥，类似于一把"钥匙"。分为：
- Sandbox Key（`sk_test_*`）：测试环境用，不会产生真实交易
- Production Key（`sk_live_*`）：生产环境用，真金白银

### Sandbox（沙箱）

模拟真实环境的测试空间。开发者可以在沙箱里测试所有功能，不会产生真实扣款。

### Magic Link（魔法链接）

一种无密码登录方式：输入邮箱 → 收到一封含有登录链接的邮件 → 点击链接即完成登录。比传统密码更安全、更方便。

### Webhook / Callback

当某个事件发生时（如交易完成、风险告警），系统主动向你的服务器发送通知。你不需要轮询，事件发生时会"推"给你。
