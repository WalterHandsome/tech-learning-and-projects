## 二、钱包生成服务

### 2.1 核心问题：Agent 的钱包从哪来？

x402 要求付款方（Agent/Client）拥有一个链上钱包，里面有 USDC。这个钱包怎么生成、私钥怎么管，是整个系统的安全地基。

### 2.2 按链生成密钥对

x402 目前支持两大链族，密钥体系完全不同：

```text
┌─────────────────────────────────────────────────────────┐
│                    EVM 链（Base / Polygon / Ethereum）    │
├─────────────────────────────────────────────────────────┤
│ 密钥算法：secp256k1 椭圆曲线                              │
│ 地址格式：0x + 20 字节十六进制（如 0x833589fC...）         │
│ 签名标准：EIP-712 结构化数据签名                           │
│ 转账机制：EIP-3009（USDC 原生支持，无需 approve）          │
│                                                         │
│ 生成流程：                                               │
│   1. 生成 32 字节随机私钥                                 │
│   2. 用 secp256k1 推导公钥                               │
│   3. 对公钥做 Keccak-256 哈希，取后 20 字节 = 地址         │
│                                                         │
│ x402 网络标识（CAIP-2 格式）：                            │
│   Base 主网：eip155:8453                                 │
│   Base Sepolia 测试网：eip155:84532                      │
│   Polygon 主网：eip155:137                               │
│   Ethereum 主网：eip155:1                                │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    Solana 链                              │
├─────────────────────────────────────────────────────────┤
│ 密钥算法：Ed25519 椭圆曲线                                │
│ 地址格式：Base58 编码（如 EPjFWdd5Aufq...）               │
│ 签名标准：Solana 原生交易签名                              │
│ 转账机制：SPL Token Program / Token2022                   │
│                                                         │
│ 生成流程：                                               │
│   1. 生成 32 字节随机种子                                 │
│   2. 用 Ed25519 推导密钥对（私钥 64 字节，公钥 32 字节）   │
│   3. 公钥的 Base58 编码 = 地址                            │
│                                                         │
│ x402 网络标识（CAIP-2 格式）：                            │
│   Solana 主网：solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp   │
│   Solana Devnet：solana:EtWTRABZaYq6iMfeYKouRu166VU2xqa1  │
└─────────────────────────────────────────────────────────┘
```


### 2.3 三种钱包管理模式对比

这是生产环境最关键的架构决策。根据行业实践，有三种模式：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ 模式一：自托管钱包（Self-Custodial / BYOW）                           │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│ 架构：用户/Agent 自己持有私钥，服务端只验证签名                        │
│                                                                      │
│ 适用场景：                                                            │
│   · 面向开发者的 API 服务（开发者自己管钱包）                          │
│   · Agent 框架集成（Agent 自带钱包能力）                               │
│   · 去中心化场景（不信任任何中间方）                                   │
│                                                                      │
│ 优点：                                                                │
│   ✓ 服务端零风险（不碰私钥）                                          │
│   ✓ 无托管合规负担                                                    │
│   ✓ 钱包可跨服务复用（一个钱包用于所有 x402 服务）                     │
│                                                                      │
│ 缺点：                                                                │
│   ✗ 用户/Agent 需要自己管理私钥（丢了就没了）                          │
│   ✗ 需要用户自己获取 USDC（入金门槛高）                               │
│   ✗ 每个 Agent 需要独立充值（资金碎片化）                              │
│                                                                      │
│ 代表方案：直接用 ethers.js / viem 生成钱包                            │
└──────────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────────┐
│ 模式二：开发者托管钱包（Developer-Controlled Wallets）                 │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│ 架构：平台通过 MPC/KMS 为每个 Agent 创建钱包，开发者通过 API 控制      │
│                                                                      │
│ 适用场景：                                                            │
│   · 平台型产品（为用户的 Agent 统一管理钱包）                          │
│   · 企业内部 Agent（公司统一管理资金和权限）                           │
│   · 需要精细控制的场景（限额、审批、审计）                             │
│                                                                      │
│ 优点：                                                                │
│   ✓ 私钥通过 MPC 分片，单点泄露不会丢失资金                           │
│   ✓ 开发者可以程序化管理（创建、充值、签名、冻结）                     │
│   ✓ 统一的资金池管理（从主钱包向 Agent 钱包分发）                      │
│                                                                      │
│ 缺点：                                                                │
│   ✗ 依赖第三方服务（Circle / Coinbase CDP / Crossmint）               │
│   ✗ 有托管责任（虽然 MPC 降低了风险）                                 │
│   ✗ 服务商锁定风险                                                    │
│                                                                      │
│ 代表方案：                                                            │
│   · Circle Developer-Controlled Wallets（MPC，多链支持）              │
│   · Coinbase CDP Wallets（与 x402 原生集成）                          │
│   · Crossmint（虚拟卡 + 钱包 + 稳定币一体化）                        │
│   · thirdweb Engine（自托管 + 托管混合模式）                          │
└──────────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────────┐
│ 模式三：硬件签名模块（HSM / Secure Enclave）                          │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│ 架构：私钥在硬件内生成且永不导出，所有签名在硬件内完成                  │
│                                                                      │
│ 适用场景：                                                            │
│   · 高安全要求（金融级、企业级）                                      │
│   · 大额资金管理（主钱包 / 资金池钱包）                               │
│   · 合规要求严格的场景                                                │
│                                                                      │
│ 优点：                                                                │
│   ✓ 最高安全等级（私钥永不离开硬件）                                  │
│   ✓ 防篡改（物理攻击会触发密钥销毁）                                  │
│   ✓ 满足金融监管要求                                                  │
│                                                                      │
│ 缺点：                                                                │
│   ✗ 成本高（硬件 + 运维）                                            │
│   ✗ 签名延迟略高（毫秒级，但对微支付可接受）                          │
│   ✗ 扩展性受限（每个 HSM 有并发上限）                                 │
│                                                                      │
│ 代表方案：                                                            │
│   · AWS CloudHSM / KMS                                               │
│   · Google Cloud KMS                                                  │
│   · Azure Key Vault（Managed HSM）                                   │
│   · OneKey 硬件签名模块（专为 x402 设计）                             │
│   · TEE 可信执行环境（如 Intel SGX / AWS Nitro Enclaves）             │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.4 生产推荐：分层钱包架构

```text
最佳实践：不同层级的钱包用不同的安全等级

┌─────────────────────────────────────────────────────────┐
│ 层级 1：主钱包（Treasury / Hot Wallet）                   │
│                                                         │
│ 安全等级：HSM / 多签（Gnosis Safe）                      │
│ 用途：持有大部分 USDC 储备                               │
│ 操作频率：低（每天 1-2 次批量分发）                       │
│ 建议：                                                   │
│   · 使用 2-of-3 或 3-of-5 多签                          │
│   · 私钥分散在不同地理位置的 HSM 中                      │
│   · 设置每日转出上限                                     │
│   · 大额转出需要人工审批                                 │
├─────────────────────────────────────────────────────────┤
│ 层级 2：运营钱包（Operations Wallet）                     │
│                                                         │
│ 安全等级：KMS / MPC                                      │
│ 用途：向 Agent 钱包分发 USDC + 支付 Gas                  │
│ 操作频率：中（每小时数十次）                              │
│ 建议：                                                   │
│   · 使用云 KMS（AWS KMS / GCP KMS）管理私钥              │
│   · 保持适量余额（够 24 小时运营即可）                    │
│   · 余额低于阈值时自动从主钱包补充                       │
│   · 所有操作记录审计日志                                 │
├─────────────────────────────────────────────────────────┤
│ 层级 3：Agent 钱包（Agent Wallets）                       │
│                                                         │
│ 安全等级：MPC / 开发者托管                               │
│ 用途：Agent 实际用来做 x402 支付的钱包                    │
│ 操作频率：高（每秒数十次签名）                            │
│ 建议：                                                   │
│   · 每个 Agent 或每个任务一个钱包                        │
│   · 钱包余额保持最小化（够完成当前任务即可）              │
│   · 任务完成后回收剩余 USDC 到运营钱包                   │
│   · 使用 Circle / CDP 的 MPC 钱包降低私钥风险            │
└─────────────────────────────────────────────────────────┘

资金流向：
  用户充值 → 主钱包 → 运营钱包 → Agent 钱包 → x402 支付
  任务完成 → Agent 钱包剩余 → 运营钱包 → 主钱包
```

### 3.1 RPC 节点接入

### 3.2 需要封装的核心链上操作

```text
EVM 链（Base / Polygon）需要封装的操作：

1. 查询 USDC 余额
   → 调用 USDC 合约的 balanceOf(address) 方法
   → Base 主网 USDC 地址：0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913
   → Base Sepolia USDC 地址：0x036CbD53842c5426634e7929541eC2318f3dCF7e
   → 注意：USDC 是 6 位小数（1 USDC = 1,000,000 最小单位）

2. USDC 转账（EIP-3009 方式 — x402 推荐）
   → 不需要先 approve 再 transferFrom
   → 付款方签名一个 TransferWithAuthorization 消息
   → Facilitator 提交这个签名到链上执行转账
   → 付款方不需要 ETH 付 Gas（Facilitator 代付）
   → 这就是 x402 "买方无 Gas" 的核心机制

3. USDC 转账（标准 transfer 方式 — 用于内部资金调拨）
   → 主钱包 → 运营钱包 → Agent 钱包 的资金分发
   → 需要发送方有 ETH 付 Gas
   → Base L2 上 Gas 极低（通常 < $0.01）

4. 查询 ETH/原生代币余额
   → Agent 钱包需要少量 ETH 用于非 x402 的链上操作
   → 运营钱包需要 ETH 用于资金分发的 Gas

5. 查询交易状态
   → 通过 txHash 查询交易是否确认
   → 用于确认资金分发和回收是否成功

Solana 链需要封装的操作：

1. 查询 USDC 余额
   → 查询 SPL Token 账户余额
   → Solana 主网 USDC Mint：EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v

2. USDC 转账
   → 通过 SPL Token Program 的 transfer 指令
   → Solana 上可以用 Kora 实现 gasless 签名（类似 EVM 的 EIP-3009）

3. 创建 Token 账户
   → Solana 上每个代币需要独立的 Token Account
   → 首次接收 USDC 前需要创建（需要少量 SOL 作为 rent）
```

### 3.3 Gas 管理策略

```text
Gas 是链上操作的"燃料费"，管理不好会导致交易失败或资金浪费。

EVM 链（Base L2）Gas 管理：

  x402 买方支付：
    → 使用 EIP-3009 时，买方不需要 Gas（Facilitator 代付）
    → 这是 x402 的核心优势之一
    → 但 Agent 钱包如果需要做非 x402 操作（如回收资金），仍需少量 ETH

  x402 卖方/Facilitator 支付：
    → Facilitator 提交链上交易需要 Gas
    → 使用 CDP Facilitator 时，Gas 由 Coinbase 承担
    → 自建 Facilitator 时，需要自己管理 Gas 钱包

  内部资金调拨：
    → 主钱包 → 运营钱包 → Agent 钱包 的转账需要 Gas
    → Base L2 上单笔 Gas 约 $0.001-0.01
    → 建议运营钱包保持 0.01-0.1 ETH 余额

Solana 链 Gas 管理：

  交易费：每笔约 0.000005 SOL（约 $0.001）
  Rent：创建 Token Account 需要约 0.002 SOL
  建议：Agent 钱包保持 0.01 SOL 余额
  Kora 方案：可实现 gasless x402 支付（类似 EVM 的 EIP-3009）
```

## 四、X402 协议对接详解

### 4.1 你是买方还是卖方？

x402 对接分两个角色，工程量完全不同：

```text
角色 A：买方（Client / Agent）—— 调用别人的付费 API

  你需要做的：
  1. 有一个带 USDC 的钱包
  2. 发 HTTP 请求 → 收到 402 → 解析支付要求
  3. 用钱包签名支付 payload
  4. 带着签名重新发请求 → 拿到数据

  工程量：小（几十行代码，用官方 SDK 即可）
  核心依赖：@x402/client（Node.js）或对应语言的 SDK

角色 B：卖方（Resource Server）—— 让别人付费调用你的 API

  你需要做的：
  1. 有一个收款钱包
  2. 在 API 前加 x402 中间件（返回 402 + 价格信息）
  3. 对接 Facilitator（验证支付 + 链上结算）
  4. 支付验证通过后返回数据

  工程量：中等（中间件配置 + Facilitator 对接）
  核心依赖：@x402/express 或 @x402/server + Facilitator

角色 C：Facilitator（支付验证与结算服务）—— 通常不需要自建

  职责：验证买方签名 → 提交链上交易 → 确认结算
  建议：直接用 Coinbase CDP Facilitator（免费 1000 笔/月）
  自建场景：需要支持 CDP 不覆盖的链、需要自定义结算逻辑
```

### 4.2 买方对接：Agent 如何自动付费

```text
完整的买方支付流程（以 Node.js + Base 链为例）：

步骤 1：准备钱包
  → 生成或导入一个 EVM 钱包（私钥 + 地址）
  → 确保钱包里有 USDC（Base 主网）
  → 不需要 ETH（EIP-3009 模式下 Facilitator 代付 Gas）

步骤 2：发起请求，处理 402 响应
  → Agent 发送普通 HTTP GET/POST 请求
  → 服务端返回 HTTP 402，响应头包含：
    X-Payment: {
      "scheme": "exact",
      "network": "eip155:8453",
      "maxAmountRequired": "1000",     // 0.001 USDC（6位小数）
      "asset": "0x833589fC...",        // USDC 合约地址
      "payTo": "0xABC...",             // 收款地址
      "maxTimeoutSeconds": 60,
      "extra": { "name": "USDC", "version": "2" }
    }

步骤 3：构造并签名支付 payload
  → 根据 402 响应中的支付要求，构造 EIP-712 结构化数据
  → 数据类型：TransferWithAuthorization（EIP-3009）
  → 包含：from（付款地址）、to（收款地址）、value（金额）、
          validAfter、validBefore、nonce
  → 用钱包私钥对这个结构化数据签名

步骤 4：带签名重新请求
  → 将签名后的 payload 放入 X-PAYMENT 请求头
  → 重新发送原始请求
  → 服务端将 payload 转发给 Facilitator 验证
  → Facilitator 验证签名 → 提交链上交易 → 确认结算
  → 服务端返回 HTTP 200 + 数据

整个过程耗时：约 200ms - 2s（取决于链上确认速度）
```

### 4.3 卖方对接：如何给 API 加付费墙

```text
以 Express.js 为例的最小实现：

步骤 1：安装依赖
  npm install @x402/express @x402/evm @x402/core

步骤 2：配置中间件
  → 指定收款地址（payTo）
  → 指定每个路由的价格
  → 指定 Facilitator URL
  → 指定网络和资产

步骤 3：中间件自动处理
  → 未付费请求 → 返回 402 + 支付要求
  → 已付费请求 → 转发 payload 给 Facilitator 验证
  → 验证通过 → 放行请求到业务逻辑
  → 验证失败 → 返回 402 要求重新支付

多网络支持（同一个端点同时接受 Base 和 Solana 支付）：
  → 在 accepts 数组中配置多个网络
  → 买方可以选择用哪条链支付
  → Facilitator 根据 network 字段路由到对应链处理
```

## 五、USDC 资金管理

### 5.1 法币到 USDC 的入金通道

Agent 钱包里的 USDC 从哪来？这是很多团队忽略的问题。

```text
方案一：交易所购买（最常见）

  流程：法币 → 交易所（Coinbase/Binance/OKX）→ 购买 USDC → 提现到主钱包
  
  优点：简单直接，费率低（0.1%-0.5%）
  缺点：需要交易所账户和 KYC，提现有延迟（分钟到小时）
  适合：初期小规模运营

方案二：OTC 大宗购买

  流程：法币 → OTC 服务商 → 大额 USDC → 主钱包
  
  优点：大额交易费率更低（可谈到 0.05%），无滑点
  缺点：需要对接 OTC 服务商，有最低交易额
  适合：日交易量 > $10,000 的场景

方案三：法币入金 API（自动化）

  流程：用户法币 → 入金 API → 自动购买 USDC → 用户的 Agent 钱包
  
  代表服务：
  · Circle Payments API（法币 → USDC，支持银行转账/信用卡）
  · MoonPay / Transak / Ramp Network（法币入金聚合器）
  · Coinbase Onramp（嵌入式法币入金组件）
  
  优点：全自动，用户体验好
  缺点：费率较高（1%-3%），需要合规对接
  适合：面向终端用户的产品

方案四：预充值模式（最简单）

  流程：平台预先购买大量 USDC → 存入主钱包 → 用户付法币给平台 →
        平台从主钱包分发 USDC 到用户的 Agent 钱包
  
  优点：用户无需接触加密货币，体验最好
  缺点：平台承担汇率风险和资金占用成本
  适合：企业级产品、不想让用户接触 crypto 的场景
```

### 5.2 资金池管理

```text
主钱包资金池的日常运营：

1. 余额监控
   → 实时监控主钱包、运营钱包、所有 Agent 钱包的 USDC 余额
   → 设置告警阈值：
     · 主钱包 < $10,000 → 提醒补充
     · 运营钱包 < $1,000 → 自动从主钱包补充
     · Agent 钱包 < $1 → 自动从运营钱包补充
   → 监控 ETH/SOL 余额（用于 Gas）

2. 自动补充（Auto Top-up）
   → 运营钱包余额低于阈值时，自动从主钱包转入
   → Agent 钱包余额低于阈值时，自动从运营钱包转入
   → 补充金额 = 预估 24 小时消耗量 × 1.5（留安全余量）

3. 资金回收（Sweep）
   → 任务完成后，Agent 钱包剩余 USDC 回收到运营钱包
   → 定期（如每天凌晨）将运营钱包超额资金回收到主钱包
   → 回收时注意保留足够的 Gas 费

4. 对账
   → 每日对账：链上余额 vs 系统记录的余额
   → 差异告警：如果链上余额 < 系统记录，可能有未记录的支出
   → 月度审计：所有资金流水的完整报告
```

## 六、签名服务（托管模式）

### 6.1 为什么需要签名服务？

```text
自托管模式：Agent 自己持有私钥，自己签名
  → 简单，但私钥分散在各个 Agent 中，管理困难

托管签名模式：Agent 不持有私钥，需要签名时请求签名服务
  → 私钥集中管理，安全性更高
  → 可以在签名前做额度验证、风控检查

签名服务的工作流程：

  Agent → 签名服务：
    "我要为这笔 x402 支付签名"
    {
      from: "0xAgentWallet",
      to: "0xMerchant",
      value: "1000",          // 0.001 USDC
      validAfter: 1234567890,
      validBefore: 1234568490,
      nonce: "0xabc..."
    }

  签名服务内部流程：
    1. 验证请求来源（Agent 身份认证）
    2. 检查 Agent 的消费额度（是否超限？）
    3. 检查收款方是否在白名单中
    4. 检查金额是否在单笔限额内
    5. 从 KMS/HSM 获取 Agent 钱包的签名能力
    6. 执行 EIP-712 签名
    7. 记录审计日志
    8. 返回签名结果

  签名服务 → Agent：
    {
      signature: "0xbfe64f8f...",
      signedAt: "2026-04-09T10:00:00Z"
    }
```

### 6.2 签名服务的安全设计

```text
关键安全原则：

1. 私钥永不导出
   → 使用 KMS/HSM，签名操作在安全硬件内完成
   → 签名服务的代码只能"请求签名"，看不到私钥本身
   → 即使签名服务被攻破，攻击者也拿不到私钥

2. 请求认证
   → 每个签名请求必须携带 Agent 的身份凭证
   → 使用 mTLS 或 JWT + API Key 双重认证
   → 防止未授权的 Agent 请求签名

3. 额度控制
   → 每个 Agent 有独立的消费额度（单笔 / 日累计 / 月累计）
   → 签名前检查额度，超限则拒绝
   → 额度可以由管理员动态调整

4. 速率限制
   → 每个 Agent 每秒最多 N 次签名请求
   → 防止被攻破的 Agent 快速耗尽资金

5. 审计日志
   → 每次签名请求都记录：谁请求的、签了什么、什么时候、结果如何
   → 日志不可篡改（写入 append-only 存储）
   → 保留至少 7 年（金融合规要求）
```

## 七、链上交易监控

### 7.1 需要监控什么？

```text
1. Agent 钱包的 USDC 转出事件
   → 每次 x402 支付都会产生一笔 USDC 转出
   → 监听 Transfer 事件，更新系统中的余额记录
   → 检测异常转出（非 x402 的意外转账）

2. Agent 钱包的 USDC 转入事件
   → 运营钱包向 Agent 钱包充值时产生
   → 确认充值到账，更新可用余额

3. 交易状态
   → 已提交但未确认的交易（pending）
   → 交易确认（confirmed）
   → 交易失败/回滚（reverted）

4. Gas 价格
   → 监控当前 Gas 价格，用于估算交易成本
   → Gas 价格异常飙升时暂停非紧急操作

5. 链的健康状态
   → RPC 节点是否正常响应
   → 区块是否正常出块
   → 链是否出现拥堵
```

### 7.2 监控实现方案

```text
方案一：轮询模式（简单但延迟高）

  每 N 秒查询一次：
  → 调用 balanceOf 查余额
  → 调用 getTransactionReceipt 查交易状态
  
  优点：实现简单
  缺点：延迟高（取决于轮询间隔），浪费 RPC 请求

方案二：WebSocket 订阅（实时但需要维护连接）

  订阅链上事件：
  → 订阅 USDC 合约的 Transfer 事件
  → 过滤 from 或 to 为我们管理的钱包地址
  → 实时收到转账通知
  
  优点：实时性好
  缺点：WebSocket 连接需要维护，断线需要重连 + 补数据

方案三：Webhook 服务（推荐）

  使用 Alchemy / QuickNode / Chainstack 的 Webhook 功能：
  → 配置监听条件（地址、事件类型）
  → 链上事件发生时，服务商主动推送到你的 Webhook 端点
  → 你只需要处理推送过来的数据
  
  优点：不需要维护连接，可靠性由服务商保证
  缺点：依赖第三方服务，有少量延迟（通常 < 5s）

生产推荐：
  · 主要用 Webhook（Alchemy Notify / QuickNode Streams）
  · 辅助用轮询（每分钟一次，用于对账和补漏）
  · 关键操作用 WebSocket（如大额转账的实时确认）
```

## 八、多链适配层

### 8.1 为什么需要多链？

```text
Base 链（EVM）：
  · x402 的"主场"（Coinbase 生态）
  · USDC 流动性最好
  · Gas 最低（L2）
  · CDP Facilitator 原生支持

Solana 链：
  · 交易确认更快（400ms vs Base 的 2s）
  · 生态中有大量 AI Agent 项目
  · thirdweb 已支持 Solana x402
  · Kora 提供 gasless 签名基础设施

同时支持两条链的价值：
  · 覆盖更多的 Agent 生态
  · 买方可以选择自己有资金的链支付
  · 降低单链故障的影响
```

### 8.2 适配层设计

```text
统一接口，屏蔽链差异：

┌─────────────────────────────────────────────────┐
│              统一钱包接口（Wallet Interface）      │
│                                                 │
│  createWallet(chain) → { address, walletId }    │
│  getBalance(address, chain) → BigNumber         │
│  transfer(from, to, amount, chain) → txHash     │
│  signPayment(payload, chain) → signature        │
│  getTransactionStatus(txHash, chain) → status   │
├─────────────────────────────────────────────────┤
│         │                         │             │
│    EVM Adapter                Solana Adapter    │
│    (ethers.js / viem)         (@solana/web3.js) │

└─────────────────────────────────────────────────┘

关键差异需要在适配层内部处理：

| 维度         | EVM (Base)              | Solana                    |
|-------------|-------------------------|---------------------------|
| 地址格式     | 0x + 40 hex chars       | Base58, 32-44 chars       |
| 签名算法     | secp256k1 + EIP-712     | Ed25519                   |
| USDC 转账   | EIP-3009 / ERC-20       | SPL Token transfer        |
| Gas 代币     | ETH                     | SOL                       |
| 确认时间     | ~2s (Base L2)           | ~400ms                    |
| 网络标识     | eip155:8453             | solana:5eykt4Us...        |
| Token 精度   | 6 decimals              | 6 decimals (相同)          |
```

### 9.2 会让你赔钱的三个配置错误

```text
这三个错误不会在启动时报错，但会在运行时默默吞掉你的钱：

错误 1：收款地址（payTo）没改
  → 开发时用的占位地址 0x0000...0001
  → 上线忘了改 → 所有收到的支付都转到了一个你不控制的地址
  → 链上不可逆，钱拿不回来
  → 防御：部署脚本中强制检查 payTo 不是零地址或占位地址

错误 2：网络 ID 和 USDC 地址不匹配
  → 网络配了主网 eip155:8453，但 USDC 地址用了测试网的
  → 或者反过来
  → 结果：Facilitator 找不到合约 → 结算失败 → 服务白给了
  → 防御：维护一个 { network → usdc_address } 的映射表，启动时校验

错误 3：汇率（rate_per_native_unit）过期
  → 如果你的定价是以 ETH 计价再转换为 USDC
  → 部署时 ETH = $3200，一个月后 ETH = $4000
  → 你的服务实际上便宜了 20%
  → 防御：定期更新汇率，或直接用 USDC 定价（推荐）
```

```text
│    └── Facilitator：x402.org（测试）/ CDP（生产）             │
└──────────────────────────────────────────────────────────────┘
```

### 10.2 这个方案的关键设计决策

```text
1. 为什么用 Circle Developer-Controlled Wallets？
   → MPC 技术：私钥被分片存储，单点泄露不会丢失资金
   → 多链支持：一个 EOA 钱包可以在所有 EVM 链上使用
   → API 驱动：创建钱包、签名、查余额都是 API 调用
   → 合规友好：Circle 是 USDC 发行方，合规基础设施完善

2. 为什么用 LangChain Agent 框架？
   → Agent 可以自主决定何时创建钱包、何时充值、何时支付
   → Tool Calling 模式天然适合 x402 的"遇到 402 就付款"逻辑
   → 可以扩展更多 Tool（如比价、选择最便宜的 API 等）

3. 为什么 Agent 不直接持有私钥？
   → Circle 的 Sign Typed Data API 在 MPC 环境中签名
   → Agent 代码中没有私钥，即使 Agent 被攻破也不会丢钱
   → 这是"签名服务"模式的最佳实践体现
```

## 十三、x402 V2 升级：从单次支付到可扩展支付基础设施

### 13.1 V1 的局限性

x402 V1 在 2025 年 5 月上线后，经过半年的真实使用暴露了几个核心问题：

```text
V1 的痛点：

1. 每次请求都要链上交易
   → Agent 调用 100 次 API = 100 笔链上交易
   → 即使 Base L2 Gas 很低，累积起来也不小
   → 高频场景下延迟不可接受

2. 没有身份层
   → Facilitator 不知道付款方是谁
   → 无法基于身份做差异化定价或信誉评估
   → 无法防止 Sybil 攻击（一个人伪装成多个付款方）

3. 单链单币
   → V1 主要支持 Base 上的 USDC
   → 不支持法币支付
   → 不支持跨链

4. SDK 耦合严重
   → 协议规范、SDK 实现、Facilitator 逻辑混在一起
   → 想支持新链或新代币需要改核心代码
```

### 13.2 V2 的核心升级（2026 年 1 月发布）

```text
升级 1：可复用会话（Reusable Sessions）

  V1：每次请求 → 签名 → 链上结算 → 返回数据
  V2：认证一次 → 建立会话 → 会话内多次请求无需重复签名

  效果：
  · 高频 API 调用场景下，延迟从秒级降到毫秒级
  · Gas 费被多次调用分摊
  · 类似 MPP 的会话模式，但原生集成在 x402 协议中

升级 2：钱包身份（Wallet-Based Identity）

  V1：付款方只是一个匿名地址
  V2：钱包地址可以关联身份信息
  · 配合 ERC-8004（Agent 链上身份标准）使用
  · 卖方可以基于身份做差异化定价
  · 支持信誉系统（历史付款记录 → 信誉分）

升级 3：多链 + 法币统一接口

  V1：主要 Base + USDC
  V2：统一支付接口，支持：
  · 多条链的稳定币（Base / Solana / Polygon / Ethereum）
  · 传统法币支付（ACH / SEPA / 信用卡网络）
  · 使用 CAIP 标准统一网络标识

升级 4：模块化插件架构

  V1：协议、SDK、Facilitator 耦合
  V2：三层分离
  · 协议规范（Protocol Spec）：定义 402 响应格式和支付流程
  · SDK 实现（@x402/core, @x402/evm, @x402/svm）：可插拔的链适配
  · Facilitator：可选择多个同时使用，支持故障转移

升级 5：自动发现（API Discovery）

  V1：买方需要预先知道 API 的价格和支付要求
  V2：标准化的发现机制
  · 类似 robots.txt，API 可以声明自己的定价信息
  · Agent 可以自动发现并比较不同 API 的价格
  · 支持动态定价（根据负载、时间等调整价格）

升级 6：动态收款方（Dynamic Payment Recipients）

  V1：收款地址在配置中写死
  V2：支持按请求路由到不同收款地址
  · 适合平台型场景（平台抽成 + 服务方收款）
  · 支持回调式分账逻辑
```

### 13.3 V2 对你的工程实现的影响

```text
如果你现在开始对接 x402，直接用 V2：

1. SDK 选择
   → 使用 @x402/core（协议核心）+ @x402/evm（EVM 链适配）
   → 如果需要 Solana，加 @x402/svm
   → V2 SDK 是模块化的，按需引入

2. 网络标识
   → 使用 CAIP-2 格式（如 eip155:8453）
   → 不要用 V1 的字符串格式（如 "base-sepolia"）

3. 会话管理
   → 高频场景优先使用 Reusable Sessions
   → 减少签名次数和链上交易

4. Facilitator
   → V2 支持同时配置多个 Facilitator
   → 建议配置 CDP + 一个备用（如 thirdweb）
```

## 十四、x402 安全漏洞与攻击面

### 14.1 已知漏洞：签名验证绕过（CVE 级别）

2026 年，安全研究人员在 x402 Facilitator 的 Python 和 Go 实现中发现了一个严重漏洞：

```text
漏洞名称：ERC-6492 签名验证绕过
影响范围：x402 Facilitator 的 verify_universal_signature 函数
严重程度：高危（可以不付钱就获取付费资源）

攻击原理：

  正常流程：
    1. 买方用私钥签名支付授权
    2. Facilitator 验证签名是否有效
    3. 验证通过 → 提交链上交易 → 结算

  攻击流程：
    1. 攻击者构造一个 ERC-6492 包装的假签名
    2. 目标地址是一个尚未部署的智能合约钱包
    3. Facilitator 检查该地址 → 没有合约代码
    4. 检查 ERC-6492 包装 → 有 factory 和 calldata → 认为是合法的未部署钱包
    5. 关键缺陷：没有验证内部签名的密码学有效性！
    6. 直接返回"验证通过"
    7. 服务端返回付费内容
    8. 但链上结算会失败（因为签名是假的）
    9. 结果：攻击者白嫖了付费资源

  为什么这个漏洞特别危险：
  · Coinbase Smart Wallet 使用反事实部署（钱包地址在合约部署前就确定）
  · 越来越多的 x402 用户会有"未部署"的钱包地址
  · 这些地址都是潜在的攻击目标

  修复方式：
  · 在返回验证通过之前，必须用 ecrecover 验证内部签名
  · 确认签名者地址与声称的付款方地址匹配
  · Coinbase 已通过 HackerOne 确认并修复
```

### 14.2 Facilitator 中心化风险

```text
当前 x402 最大的架构风险：所有支付都经过一个中心化的 Facilitator

三种失败模式：

1. 伪造结算（Fabrication）
   → 被攻破的 Facilitator 可以声称"支付已结算"但实际没有
   → 服务端信任 Facilitator 的响应，执行了计算
   → 结果：服务方白干活，收不到钱

2. 审查支付（Censorship）
   → Facilitator 可以拒绝验证或结算合法的支付
   → 由于它是唯一的结算路径，被审查的客户端完全无法使用服务

3. 服务宕机（Downtime）
   → Facilitator 下线 → 所有 x402 服务停止接受支付
   → 不是你的服务有问题，不是链有问题，就是一个 HTTP 端点挂了

核心问题：
  · 服务端收到 Facilitator 的结算响应后，没有做链上验证
  · 没有密码学证明结算确实发生了
  · 没有交易哈希查询
  · Facilitator 说"结算成功"，服务端就信了

对比其他系统：
  · 闪电网络：每一跳都有密码学原像证明支付到达
  · OpenGSN 中继器：必须提交链上收据，否则保证金被没收
  · x402 Facilitator：既没有密码学证明，也没有经济惩罚
```

### 14.3 去中心化 Facilitator 的演进路线

```text
阶段 1：验证 Facilitator 的声明（当前可做）
  → Facilitator 结算后，服务端自己查链上收据
  → 如果链上没有对应交易 → 拒绝执行
  → 增加一次 eth_call，延迟约 50-100ms
  → 关闭"伪造结算"攻击向量

阶段 2：本地验证签名（减少依赖）
  → 服务端自己验证 EIP-3009 签名的有效性
  → 自己检查付款方余额
  → Facilitator 只负责提交链上交易（结算）
  → 关闭"审查支付"攻击向量

阶段 3：自行结算（完全去中心化）
  → 服务端直接提交 transferWithAuthorization 交易
  → 不再需要 Facilitator
  → Base L2 上 Gas 费 < $0.01，经济上完全可行
  → 关闭"宕机"攻击向量

阶段 4：链上强制执行（终极方案）
  → 报价注册表上链（定价透明可审计）
  → 智能合约强制执行分账逻辑
  → 如果收了钱不执行服务 → 质押金被罚没
  → 从"信任 Facilitator"变为"作恶在经济上不合理"
```

### 14.4 应用层安全清单

```text
除了协议层漏洞，应用层也有独特的攻击面：

1. 私钥管理
   ✗ 不要把私钥硬编码在 .env 文件中（开发可以，生产绝对不行）
   ✗ 不要每个服务器一个私钥（密钥蔓延）
   ✓ 使用 KMS/MPC/HSM 管理私钥
   ✓ 私钥永不以明文形式出现在应用代码中

2. 重放攻击防护
   ✓ x402 协议内置了 nonce 和 validBefore/validAfter 时间窗口
   ✓ 但要确保你的实现正确检查了这些字段
   ✓ 特别注意：不要接受过期的支付授权

3. 前端运行（Frontrunning）防护
   → EIP-3009 的 transferWithAuthorization 指定了 from 和 to
   → 攻击者无法重定向资金
   → 但在公共 mempool 中，交易内容是可见的
   → 如果需要额外保护，使用私有 RPC（如 Flashbots Protect）

4. 跨链重放
   → 同一个签名在不同链上可能有效
   → x402 V2 使用 CAIP-2 网络标识 + EIP-712 domain separator 防止
   → 确保你的 domain separator 包含正确的 chainId

5. 价格操纵
   → 如果定价基于 ETH/USDC 汇率，汇率波动会影响实际价格
   → 建议直接用 USDC 定价（避免汇率风险）
   → 如果必须用 ETH 定价，设置合理的 markup_bps 覆盖波动
```

## 十五、ERC-8004：x402 的身份层补充

### 15.1 x402 缺什么？

```text
x402 解决了"怎么付钱"，但没有解决"谁在付钱"。

问题场景：
  · 服务方收到一笔 x402 支付，只知道付款地址是 0xABC...
  · 不知道这个地址背后是谁（人？Agent？恶意脚本？）
  · 不知道这个地址的历史信誉（是否有过恶意行为？）
  · 无法基于身份做差异化定价（老客户优惠？高信誉免审核？）

这就是 ERC-8004 要解决的问题。
```

### 15.2 ERC-8004 是什么？

```text
ERC-8004 是 2026 年 1 月上线的 Ethereum 标准，为 AI Agent 提供链上身份。

三个核心注册表：

1. 身份注册表（Identity Registry）
   → Agent 在链上注册自己的身份
   → 包含：名称、描述、服务端点、能力声明
   → 类似"Agent 的名片"

2. 信誉注册表（Reputation Registry）
   → 记录 Agent 的历史行为评分
   → 其他 Agent 或用户可以给评价
   → 类似"Agent 的信用报告"

3. 验证注册表（Validation Registry）
   → 第三方验证机构可以为 Agent 背书
   → 类似"Agent 的资质证书"

已部署的链：
  Ethereum 主网、BNB Chain、Base、Polygon、Arbitrum 等 17+ 条 EVM 链
  The Graph 提供了跨链索引子图
```

### 15.3 ERC-8004 + x402 = 完整的 Agent 经济基础设施

```text
组合使用的场景：

1. 基于信誉的差异化定价
   → Agent 调用付费 API → 服务方查询 Agent 的 ERC-8004 信誉分
   → 高信誉 Agent → 优惠价格 / 更高额度
   → 低信誉或无信誉 Agent → 标准价格 / 需要预付

2. 基于身份的访问控制
   → 某些高价值 API 只对已注册身份的 Agent 开放
   → 未注册的匿名地址 → 拒绝服务或限制功能

3. Agent 间信任建立
   → Agent A 要调用 Agent B 的服务
   → Agent A 查询 Agent B 的 ERC-8004 身份和信誉
   → 信誉足够 → 直接交易
   → 信誉不足 → 要求预付或拒绝

类比：
  x402 = 货币（解决"怎么付"）
  ERC-8004 = 护照（解决"你是谁"）
  两者结合 = Agent 可以在互联网上自主地、有信任地进行经济活动
```
