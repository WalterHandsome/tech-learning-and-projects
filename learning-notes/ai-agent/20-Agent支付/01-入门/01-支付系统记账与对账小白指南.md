## 一、为什么支付系统需要"记账"？

### 1.1 一个没有记账的灾难故事

想象你开了一家奶茶店，每天收很多钱。你不记账，只看收银机里有多少现金。

```
周一：收银机里有 $1000
周二：收银机里有 $800

问题来了：
  → 少了 $200，是卖了 $200 的奶茶？
  → 还是员工偷了 $200？
  → 还是你自己拿了 $200 去买原料？
  → 你不知道。因为你没记账。
```

支付系统也一样。如果只记"余额是多少"，不记"每一笔钱从哪来、到哪去"，出了问题就无法排查。

### 1.2 支付系统记账的目标

```
目标 1：知道每一分钱在哪里
  → 用户卡上冻结了多少？平台收了多少手续费？链上转了多少？

目标 2：出了问题能查到原因
  → 用户说"我被多扣了 $5" → 你能查到这 $5 是哪笔交易产生的

目标 3：满足监管要求
  → 金融监管要求所有资金流动必须有记录，可审计，可追溯

目标 4：防止重复扣款
  → 网络抖动导致请求重试，不能扣两次钱
```

## 二、什么是复式记账？（从零开始）

### 2.1 先理解"单式记账"

你的银行 APP 就是单式记账：

```
日期        描述          金额
─────────────────────────────
5/1    工资入账        +$5000
5/2    房租            -$1500
5/3    买咖啡          -$5
5/4    转账给朋友      -$200
─────────────────────────────
余额                   $3295
```

**问题**：你知道钱少了，但不知道钱去了哪个"口袋"。转给朋友的 $200，朋友收到了吗？房租 $1500，房东收到了吗？你只有自己这一边的记录。

### 2.2 复式记账：每笔钱都有"来"和"去"

复式记账的核心规则只有一条：

> **每笔钱的流动，必须同时记录"从哪来"和"到哪去"，两边金额必须相等。**

用生活例子：

```
你转 $200 给朋友：

  单式记账（只记一边）：
    你的账户：-$200

  复式记账（记两边）：
    你的账户：-$200（钱出去了）
    朋友的账户：+$200（钱进来了）
    
    验证：$200 = $200 ✓ 平衡！
```

```
你用信用卡买了 $5 的咖啡：

  复式记账：
    你的信用卡账户：-$5（你的钱少了）
    咖啡店的收入账户：+$5（咖啡店的钱多了）
    
    验证：$5 = $5 ✓ 平衡！
```

### 2.3 "借方"和"贷方"是什么？

很多人被"借方（Debit）"和"贷方（Credit）"这两个词吓到。其实很简单：

```
借方（Debit）= 钱进入这个账户 = "进"
贷方（Credit）= 钱离开这个账户 = "出"

例子：你转 $200 给朋友

  借方：朋友的账户 +$200（朋友的钱多了，钱"进"了朋友的口袋）
  贷方：你的账户 -$200（你的钱少了，钱"出"了你的口袋）
```

**记住一个口诀：借方 = 进，贷方 = 出，进出必须相等。**

### 2.4 为什么支付公司都用复式记账？

```
原因 1：自动查错
  → 如果某笔交易只记了"进"没记"出"，借贷不平衡
  → 系统立刻告警："有 bug！"
  → 单式记账发现不了这种错误

原因 2：防止重复扣款
  → 每笔交易有唯一的 idempotency_key（幂等键）
  → 同一个 key 只能记一次账
  → 网络重试不会导致重复记账

原因 3：可审计
  → 监管机构来查账，你能展示每一分钱的完整流向
  → "这 $100 从用户卡上来，$95 给了商户，$5 是我们的手续费"

原因 4：可对账
  → 你的记录说"转了 $100 给商户"
  → 商户的记录说"收到了 $100"
  → 两边一比对，对上了 → 没问题
  → 对不上 → 有问题，要查
```

行业标准：Stripe、Square、支付宝、微信支付、所有银行，全部使用复式记账。这不是"最佳实践"，而是**唯一实践**——没有任何正规支付公司用单式记账。

## 三、用你的 AI Payment 平台举例

### 3.1 平台里有哪些"账户"（钱罐子）？

```
把每个账户想象成一个贴了标签的钱罐子：

用户侧：
  🏦 member_available    用户的可用余额（卡上还能花的钱）
  🔒 member_held         用户的冻结金额（已经被预授权冻结的钱）

平台侧：
  💰 system_revenue      平台的收入（手续费、markup）
  📦 system_settlement   待结算（已经消费但还没打到银行的钱）

X402 专属：
  ⛓️ x402_wallet_frozen  大钱包的冻结额（签名已发但链上还没转出的 USDC）

Provider 侧：
  🏪 vcn_provider        VCN 供应商应付
  🔗 nt_provider         Network Token 供应商应付
```

### 3.2 一笔完整交易的记账过程

场景：Agent 用 X402 支付了 $1 的 API 调用费，markup 5%（$0.05）

```
═══ 步骤 1：创建 Token（冻结 $1.05）═══

  用户卡上冻结 $1.05（$1 本金 + $0.05 markup）

  借方：🔒 member_held        +$1.05（冻结增加）
  贷方：🏦 member_available    -$1.05（可用减少）
  
  同时，大钱包冻结 $1 的 USDC：
  借方：⛓️ x402_wallet_frozen  +$1.00（USDC 冻结增加）
  贷方：（无，这是链上资产的内部标记）

  验证：$1.05 = $1.05 ✓

═══ 步骤 2：链上确认（Agent 真的花了 $1）═══

  Alchemy Webhook 通知：链上转出了 1 USDC

  借方：📦 system_settlement   +$1.00（待结算增加）
  贷方：🔒 member_held         -$1.00（冻结减少）
  
  释放 USDC 冻结：
  贷方：⛓️ x402_wallet_frozen  -$1.00（USDC 冻结释放）

  验证：$1.00 = $1.00 ✓

═══ 步骤 3：分配 markup $0.05 ═══

  借方：💰 system_revenue      +$0.05（平台收入增加）
  贷方：🔒 member_held         -$0.05（冻结清零）

  验证：$0.05 = $0.05 ✓

═══ 全程验证 ═══

  总借方：$1.05 + $1.00 + $0.05 = $2.10
  总贷方：$1.05 + $1.00 + $0.05 = $2.10 ✓ 完美平衡

  member_held 变化：+$1.05 → -$1.00 → -$0.05 = $0（清零 ✓）
  member_available 变化：-$1.05（用户实际被扣 $1.05 ✓）
```

### 3.3 如果交易取消了呢？

```
场景：创建了 Token 但 Agent 没用，签名过期了

═══ 步骤 1：创建 Token（冻结 $1.05）═══
  （同上）

═══ 步骤 2：签名过期，释放冻结 ═══

  情况 A：markup = 0，全额释放
    借方：🏦 member_available    +$1.05（可用余额恢复）
    贷方：🔒 member_held         -$1.05（冻结释放）
    验证：$1.05 = $1.05 ✓
    → 用户一分钱没花

  情况 B：markup > 0，capture markup 部分
    借方：🏦 member_available    +$1.00（本金退回）
    借方：💰 system_revenue      +$0.05（markup 归平台）
    贷方：🔒 member_held         -$1.05（冻结释放）
    验证：$1.00 + $0.05 = $1.05 ✓
    → 用户被扣了 $0.05 的 markup（即使没消费）
```

## 四、什么是对账？

### 4.1 生活中的对账

```
你和朋友 AA 制吃饭，你先付了 $100。

你的记录：我付了 $100，朋友欠我 $50
朋友的记录：我欠他 $50

对账 = 比对两边的记录：
  你说 $50，朋友也说 $50 → 对上了 ✓
  你说 $50，朋友说 $40 → 对不上 ✗ → 要查原因
```

### 4.2 支付系统的对账

你的 AI Payment 平台有**三个独立的记录系统**，它们各自记各自的：

```
系统 A：你的内部账本
  → "我冻结了用户 $105，capture 了 $100，释放了 $5"

系统 B：Evo Payment（卡网络网关）
  → "我收到了预授权 $105，capture 了 $100，cancel 了 $5"

系统 C：区块链（X402 专属）
  → "大钱包转出了 100 USDC 到地址 0xABC..."

对账 = 确认 A、B、C 三方的记录一致
```

### 4.3 三方对账（Three-Way Reconciliation）

这是支付行业的标准做法。你的平台需要做的对账：

```
第一层：自己和自己对（自洽性）
  → 我的借方总额 = 贷方总额吗？
  → 所有 ACTIVE Token 的冻结总额 = wallet_ledger.total_frozen 吗？
  → 如果不等 → 我自己的系统有 bug

第二层：自己和 Evo 对（内部 vs 卡网络）
  → 我记录的每笔 preauth，Evo 也有吗？金额一样吗？
  → 我记录的每笔 capture，Evo 也有吗？金额一样吗？
  → 如果不一样 → 要么我的记录错了，要么 Evo 的记录错了

第三层：自己和链上对（内部 vs 区块链）
  → 我记录的每笔 CONFIRMED 交易，链上有对应的 USDC 转账吗？
  → 链上的每笔转出，我都有记录吗？
  → 如果链上有转出但我没记录 → 可能被盗了！

第四层：Evo 和银行对（卡网络 vs 实际入账）
  → Evo 说结算了 $10000 给我们
  → 银行账户实际收到 $10000 了吗？
  → 如果不一样 → 联系 Evo 或银行
```

来源：[Razorpay: What is Three-Way Matching](https://razorpay.com/learn/what-is-three-way-matching/) (Content was rephrased for compliance with licensing restrictions)

## 五、AI Agent 支付和传统支付有什么不同？

### 5.1 传统支付（人类购物）

```
你在淘宝买东西：
  1. 你选商品，点"购买"
  2. 你输入密码 / 刷脸确认
  3. 支付宝扣你的钱
  4. 商家发货
  5. 你确认收货
  6. 支付宝把钱给商家

每一步都有人参与。出了问题，你可以打客服电话。
```

### 5.2 AI Agent 支付（机器购物）

```
Agent 帮你调用付费 API：
  1. Agent 请求 API → 收到 "402 要付钱"
  2. Agent 需要付 $0.003（三厘钱）
  3. ??? Agent 没有信用卡，没有支付宝，没有手指输密码
  4. 而且 Agent 可能一秒钟调用 100 次
  5. 每次只花 $0.003，信用卡最低手续费 $0.30，倒贴 100 倍

这就是为什么需要你们的 AI Payment 平台。
```

### 5.3 AI Payment 平台解决了什么？

```
问题 1：Agent 没有支付能力
  → 解决：平台给 Agent 发"令牌"（Token），Agent 拿令牌去付钱

问题 2：微支付手续费太高
  → 解决：X402 链上支付，手续费 $0.001（信用卡的 1/300）

问题 3：Agent 可能被黑客控制乱花钱
  → 解决：令牌有限额、有时效、有商户限制（策略引擎）

问题 4：怎么从用户卡上扣钱？
  → 解决：先预授权冻结（freeze），确认消费后再扣（capture）

问题 5：怎么确保每分钱都对得上？
  → 解决：复式记账 + 三方对账（就是这篇文档讲的内容）
```

## 六、行业主流产品怎么做的？

### 6.1 产品对比

| 产品 | 定位 | 记账方式 | 对账方式 | 支付轨道 |
|------|------|---------|---------|---------|
| **Stripe Issuing** | 给 Agent 发虚拟卡 | Stripe 内部复式账本 | Stripe Dashboard + API | 卡网络（Visa/MC） |
| **Crossmint** | Agent 全栈支付 | 内部账本 + 链上记录 | 链上自动 + 手动 | 虚拟卡 + 稳定币 |
| **x402 原生** | 链上微支付 | 链上记录即账本 | 链上自动验证 | 仅稳定币 |
| **Stripe MPP** | 会话式微支付 | Stripe 账本 + 链上 | Stripe 自动 | 稳定币（Tempo 链） |
| **你的平台** | 多轨道中间层 | MongoDB 内部账本 | 四层校验（待实现 T14） | VCN + NT + X402 |

### 6.2 Stripe 怎么做记账和对账？

```
Stripe 的做法（行业标杆）：

记账：
  → 每笔交易生成 Balance Transaction（余额交易记录）
  → 包含：金额、手续费、净额、来源、时间
  → 这就是复式记账的一种实现
  → 所有记录不可修改（immutable）

对账：
  → Stripe 自动对账：内部记录 vs 卡网络记录
  → 提供 Payout Reconciliation API：你可以查每笔入账的明细
  → 提供 Sigma（SQL 查询）：你可以自己写查询做对账
  → 争议（Dispute）自动处理：用户投诉时自动冻结资金

你的平台可以学习的：
  → Balance Transaction 的设计思路 → 你的 ap_token_transactions
  → Payout Reconciliation → 你的 L4 对账
  → 不可修改的记录 → 你的 ledger_entries 只 INSERT 不 UPDATE
```

### 6.3 Crossmint 怎么做？

```
Crossmint 的做法（Agent 支付新锐）：

架构：四层支付栈
  Layer 1：钱包层（Agent 的钱在哪里）
    → 稳定币钱包（USDC）+ 虚拟卡
  Layer 2：入金层（钱怎么进来）
    → 法币 → 稳定币的 onramp
  Layer 3：编排层（交易怎么路由）
    → 根据金额和场景自动选择最优支付轨道
  Layer 4：卡网络层（传统商户怎么付）
    → 虚拟卡走 Visa/MC 网络

对账：
  → 链上交易：区块链自动验证（不可篡改）
  → 卡网络交易：和 Visa/MC 的结算报告对账
  → 两条轨道独立对账，最终汇总

你的平台和 Crossmint 的相似之处：
  → 都是多轨道（链上 + 卡网络）
  → 都需要两套对账逻辑
  → 你的 X402 ≈ Crossmint 的稳定币钱包
  → 你的 VCN/NT ≈ Crossmint 的虚拟卡
```

## 八、你的平台的对账方案总结

```
四层校验（从内到外）：

L1 自洽性（每小时）：自己的账本借贷平衡吗？冻结总额对吗？
L2 内部 vs Evo（每日）：我的 preauth/capture/cancel 和 Evo 一致吗？
L3 内部 vs 链上（每小时）：我的 CONFIRMED 记录和链上转账一致吗？
L4 Evo vs 银行（每周）：Evo 结算的钱真的到银行了吗？

异常升级：
  差异 < $0.01 → 自动忽略（精度取整）
  差异 $0.01~$100 → Slack 告警 + 冻结相关 Token
  差异 > $100 → PagerDuty + 暂停服务 + 人工介入
```

| 术语 | 英文 | 一句话解释 |
|------|------|-----------|
| 预授权 | Preauth / Authorization | 先冻结用户卡上的钱，还没真扣 |
| 扣款 | Capture | 从冻结的钱里真正扣掉 |
| 释放 | Cancel / Void | 把冻结的钱还给用户 |
| 复式记账 | Double-Entry Bookkeeping | 每笔钱记"从哪来"和"到哪去"两条 |
| 借方 | Debit | 钱进入这个账户 |
| 贷方 | Credit | 钱离开这个账户 |
| 对账 | Reconciliation | 比对两个系统的记录是否一致 |
| 幂等 | Idempotency | 同一个操作执行多次，结果和执行一次一样 |
| Markup | Markup | 平台加价（手续费的一种） |
| Gateway Token | Gateway Token | 绑卡时 Evo 返回的令牌，代替真实卡号 |
| Facilitator | Facilitator | x402 协议中负责链上结算的中间方（Coinbase） |
| EIP-712 | EIP-712 | 以太坊的签名标准，用于 x402 支付签名 |
