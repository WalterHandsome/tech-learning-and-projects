## 1. Transformer 架构

Transformer 是现代大语言模型的基础架构，由 Vaswani 等人在 2017 年论文 *"Attention Is All You Need"* 中提出。其核心机制是**自注意力（Self-Attention）**，允许模型在处理每个 token 时关注输入序列中的所有其他 token。

```text
输入文本 → Tokenization → Embedding + Positional Encoding → [N × Transformer Blocks] → 输出

Encoder Block（BERT 类）:
  Multi-Head Self-Attention → Add & Norm → Feed Forward → Add & Norm

Decoder Block（GPT 类）:
  Masked Self-Attention → Cross-Attention（可选）→ Feed Forward → 输出概率分布

现代 LLM 几乎全部采用 Decoder-Only 架构（GPT、Claude、Llama、Qwen 等）
```

### 核心：缩放点积注意力

```python
# Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) * V
# Q = Query（我在找什么）
# K = Key（我有什么）
# V = Value（我的内容是什么）

import torch
import torch.nn.functional as F

def scaled_dot_product_attention(Q, K, V, mask=None):
    """缩放点积注意力 — Transformer 的核心计算"""
    d_k = Q.size(-1)
    # 计算注意力分数
    scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_k ** 0.5)
    # 可选：因果掩码（Decoder 中防止看到未来 token）
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))
    # softmax 归一化为概率分布
    weights = F.softmax(scores, dim=-1)
    # 加权求和
    return torch.matmul(weights, V), weights
```

## 3. 推理模型 vs 通用模型（2025-2026 新范式）

2024 年底 OpenAI 发布 o1 标志着**推理模型（Reasoning Model）**成为新范式。推理模型在回答前会进行"深度思考"，通过推理时计算（inference-time compute）来提升复杂问题的解决能力。

### 核心区别

```text
通用模型（Standard Model）:
  用户提问 → 模型直接生成回答
  特点：快速、低成本、适合大多数任务

推理模型（Reasoning Model）:
  用户提问 → 模型内部思考链（可能数千 tokens）→ 生成回答
  特点：慢、高成本、复杂推理/数学/编码显著更强
```

### 主要推理模型

| 模型 | 厂商 | 特点 |
| --- | --- | --- |
| o3 | OpenAI | 最强推理，支持 reasoning_effort 参数（low/medium/high） |
| o4-mini | OpenAI | 轻量推理，成本仅 o3 的 1/2 |
| DeepSeek R1 | DeepSeek | 开源推理模型，MIT 许可，可本地部署 |
| Claude Extended Thinking | Anthropic | Claude 的推理模式，支持 thinking budget 控制 |
| Gemini 3.1 Pro（思考模式） | Google | 内置推理能力，2M 上下文下推理 |

### 何时使用推理模型？

```text
✅ 适合推理模型的场景：
  - 复杂数学/逻辑推理
  - 多步骤代码调试与架构设计
  - 科学问题分析
  - 需要深度分析的决策

❌ 不适合推理模型的场景：
  - 简单问答、分类、摘要
  - 高吞吐低延迟需求
  - 成本敏感的批量处理
  - 创意写作（推理模型倾向过度分析）
```

### 成本影响

推理模型的"思考"过程会消耗大量 tokens。一个简单问题，通用模型可能用 200 tokens 回答，推理模型可能用 2000+ tokens（其中 1800 是思考过程）。因此：

- **推理 tokens 通常按输出价格计费**，成本可能是通用模型的 5-10 倍
- 合理使用 `reasoning_effort` 参数控制思考深度
- 简单任务用通用模型，复杂任务才用推理模型

## 4. Token 与上下文窗口

### Token 基础

```python
import tiktoken

# GPT 系列使用 tiktoken（BPE 分词）
enc = tiktoken.encoding_for_model("gpt-4o")
text = "AI Agent 是一个能够自主决策的智能系统"
tokens = enc.encode(text)
print(f"文本: {text}")
print(f"Token 数: {len(tokens)}")  # 中文通常 1 字 ≈ 1-2 tokens
print(f"Token IDs: {tokens}")
print(f"解码: {enc.decode(tokens)}")

# 不同语言的 token 效率差异很大
# 英文: 1 word ≈ 1-1.5 tokens
# 中文: 1 字 ≈ 1-2 tokens
# 代码: 变化大，取决于变量名和语法
```

### 上下文窗口 vs 有效上下文

> ⚠️ **重要提醒**：上下文窗口大小 ≠ 有效利用能力。研究表明：
>
> - 模型在上下文的**开头和结尾**表现最好（"Lost in the Middle" 现象）
> - 超过 100K tokens 后，检索准确率开始下降
> - 1M+ 上下文适合"大海捞针"式检索，但不适合需要全局理解的任务
> - 实际应用中，RAG + 适度上下文 通常优于 暴力塞满上下文

```python
# 上下文窗口计算
# 总上下文 = 系统提示 + 历史消息 + 用户输入 + 工具结果 + 模型输出
# 示例：Claude Sonnet 4.6
max_context = 1_000_000  # 1M tokens

user_input = 5_000
reserved_output = 8_192
available_for_tools = max_context - system_prompt - history - user_input - reserved_output
```

## 5. 采样参数

```python
from openai import OpenAI
client = OpenAI()

# Temperature: 控制随机性（0=确定性，2=高随机性）
# Top-P: 核采样，从概率累积前 P 的 token 中采样
response = client.chat.completions.create(
    model="gpt-5.2",
    messages=[{"role": "user", "content": "写一首关于AI的诗"}],
    temperature=0.7,   # 创意任务用 0.7-1.0
    top_p=0.9,         # 通常与 temperature 二选一调整
    max_tokens=500,
    frequency_penalty=0.5,  # 减少重复（0-2）
    presence_penalty=0.3,   # 鼓励新话题（0-2）
)
```

### 不同场景推荐参数

| 场景 | temperature | top_p | 说明 |
| --- | --- | --- | --- |
| 代码生成 | 0 | 1 | 确定性输出，避免幻觉 |
| 数据提取/分类 | 0 | 1 | 需要精确、可重复的结果 |
| 对话聊天 | 0.7 | 0.9 | 自然流畅但不过于随机 |
| 创意写作 | 0.9 | 0.95 | 高创造性，多样化表达 |
| Agent 工具调用 | 0 | 1 | 工具参数必须精确 |

> **注意**：推理模型（o3、o4-mini）通常不支持 temperature 参数，因为其内部思考过程已经包含了探索性。Claude 的 Extended Thinking 模式下也建议使用 temperature=1。

## 7. Prompt Caching（2025-2026 关键特性）

Prompt Caching 是 2025-2026 年最重要的成本优化技术之一，对 Agent 架构设计产生了深远影响。

### 核心原理

```text
传统调用：
  每次请求 → 完整处理所有输入 tokens → 按输入价格计费

Prompt Caching：
  首次请求 → 处理并缓存前缀 → 按输入价格计费
  后续请求 → 命中缓存前缀 → 缓存部分享受 90% 折扣，仅新增部分按原价

关键：缓存基于前缀匹配（prefix matching）
  [系统提示 | 工具定义 | 历史消息 | 新消息]
   ←——— 可缓存的前缀 ———→  ← 新增 →
```

### 成本影响

| 提供商 | 缓存写入 | 缓存读取（命中） | 节省比例 |
| --- | --- | --- | --- |
| Anthropic | 1.25× 输入价格 | 0.1× 输入价格 | 90% |
| OpenAI | 1× 输入价格 | 0.5× 输入价格 | 50% |
| Google | 1× 输入价格 | 0.25× 输入价格 | 75% |

### API 示例

```python
from anthropic import Anthropic

client = Anthropic()

# Anthropic: 显式缓存控制
response = client.messages.create(
    model="claude-sonnet-4-6-20260401",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "你是一个代码审查助手...(很长的系统提示)...",
            "cache_control": {"type": "ephemeral"}  # 标记为可缓存
        }
    ],
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": "大量代码上下文...",
                    "cache_control": {"type": "ephemeral"}  # 代码上下文也缓存
                },
                {
                    "type": "text",
                    "text": "请审查第 42 行的变更"  # 只有这部分是新的
                }
            ]
        }
    ],
)

# 查看缓存命中情况
print(f"缓存写入 tokens: {response.usage.cache_creation_input_tokens}")
print(f"缓存命中 tokens: {response.usage.cache_read_input_tokens}")
print(f"未缓存 tokens: {response.usage.input_tokens}")
```

```python
from openai import OpenAI

client = OpenAI()

# OpenAI: 自动缓存（无需显式标记）
# 相同前缀的请求自动命中缓存
response = client.chat.completions.create(
    model="gpt-5.2",
    messages=[
        {"role": "system", "content": "很长的系统提示..."},  # 自动缓存
        {"role": "user", "content": "新的用户问题"}
    ],
)
# 通过 usage.prompt_tokens_details.cached_tokens 查看命中
```

### 对 Agent 架构的影响

Prompt Caching 深刻改变了 Agent 的设计思路：

- **Claude Code 的经济学**：Claude Code 大量使用 Prompt Caching，将整个代码库作为缓存前缀，每次工具调用只需为新增的少量 tokens 付费
- **系统提示设计**：将不变的内容（工具定义、规则、示例）放在前面，变化的内容放在后面
- **对话历史管理**：保持消息顺序稳定，避免打乱缓存前缀
- **批量处理**：相同前缀的多个请求可以共享缓存

## 8. Structured Output（结构化输出）

结构化输出确保模型返回符合预定义 schema 的 JSON，是 Agent 工具调用和数据提取的基础。

### JSON Mode

```python
from openai import OpenAI
client = OpenAI()

# OpenAI JSON Mode
response = client.chat.completions.create(
    model="gpt-5.2",
    response_format={"type": "json_object"},
    messages=[
        {"role": "system", "content": "以 JSON 格式返回分析结果"},
        {"role": "user", "content": "分析这段文本的情感倾向：'今天天气真好'"}
    ],
)
# 返回: {"sentiment": "positive", "confidence": 0.95}
```

### Schema 强制（Structured Outputs）

```python
from pydantic import BaseModel
from openai import OpenAI

client = OpenAI()

# 使用 Pydantic 定义严格 schema
class SentimentAnalysis(BaseModel):
    sentiment: str  # "positive", "negative", "neutral"
    confidence: float
    key_phrases: list[str]
    reasoning: str

response = client.beta.chat.completions.parse(
    model="gpt-5.2",
    messages=[
        {"role": "user", "content": "分析：'这个产品质量不错但价格偏高'"}
    ],
    response_format=SentimentAnalysis,
)

result = response.choices[0].message.parsed
print(f"情感: {result.sentiment}")       # "mixed" 或 "neutral"
print(f"置信度: {result.confidence}")     # 0.82
print(f"关键短语: {result.key_phrases}")  # ["质量不错", "价格偏高"]
```

```python
import anthropic

client = anthropic.Anthropic()

                        "type": "string",
```

> **关键区别**：
> - **JSON Mode**：模型输出 JSON 格式，但不保证 schema 一致性
> - **Structured Output**：严格遵循预定义 schema，100% 类型安全
> - **Tool Use**：模型决定调用哪个工具并生成参数，是 Agent 的核心机制

## 9. Extended Thinking / 推理预算

Extended Thinking 允许开发者控制模型的"思考深度"，在质量和成本之间取得平衡。

### Claude Extended Thinking

```python
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-opus-4-6-20260401",
    max_tokens=16000,
    thinking={
        "type": "enabled",
        "budget_tokens": 10000  # 最多用 10K tokens 思考
    },
    messages=[{
        "role": "user",
        "content": "设计一个分布式任务调度系统的架构，考虑容错和扩展性"
    }],
)

# 响应包含 thinking blocks 和 text blocks
for block in response.content:
    if block.type == "thinking":
        print(f"思考过程（{len(block.thinking)} 字符）:")
        print(block.thinking[:200] + "...")
    elif block.type == "text":
        print(f"\n最终回答:")
        print(block.text)
```

```python
    messages=[{
        "role": "user",
        "content": "证明：对于所有正整数 n，n^3 - n 能被 6 整除"
    }],
)

# reasoning_effort 影响：
# low    → 快速回答，少量思考，适合简单问题
# medium → 平衡模式，适合大多数任务
# high   → 深度思考，适合复杂数学/编码问题
```

### 推理预算策略

```text
任务复杂度评估 → 选择推理预算：

简单任务（分类、摘要、格式转换）:
  → 使用通用模型（Sonnet/GPT-5.2），无需推理预算
  → 成本: ~$0.01/请求

中等任务（代码生成、分析报告）:
  → Extended Thinking budget_tokens=5000 或 reasoning_effort="medium"
  → 成本: ~$0.05/请求

复杂任务（架构设计、数学证明、多步调试）:
  → Extended Thinking budget_tokens=30000 或 reasoning_effort="high"
  → 成本: ~$0.20/请求
```

## 10. Batch API 与成本优化

### Batch API

Batch API 允许批量提交请求，以更低的价格异步处理，适合非实时场景。

```python
from openai import OpenAI

client = OpenAI()

# 1. 准备批量请求（JSONL 格式）
import json

requests = []
for i, text in enumerate(texts_to_analyze):
    requests.append({
        "custom_id": f"request-{i}",
        "method": "POST",
        "url": "/v1/chat/completions",
        "body": {
            "model": "gpt-5.2",
            "messages": [
                {"role": "system", "content": "分析文本情感"},
                {"role": "user", "content": text}
            ],
            "max_tokens": 100
        }
    })

# 写入 JSONL 文件
with open("batch_input.jsonl", "w") as f:
    for req in requests:
        f.write(json.dumps(req) + "\n")

# 2. 上传并创建批处理任务
batch_input_file = client.files.create(
    file=open("batch_input.jsonl", "rb"),
    purpose="batch"
)

batch = client.batches.create(
    input_file_id=batch_input_file.id,
    endpoint="/v1/chat/completions",
    completion_window="24h"  # 24 小时内完成
)

# 3. 价格优势：Batch API 享受 50% 折扣
# GPT-5.2 实时: $1.75/M 输入, $14/M 输出
# GPT-5.2 Batch: $0.875/M 输入, $7/M 输出
```

### 成本优化策略总览

```text
成本优化金字塔（从上到下优先级递减）：

1. 选对模型
   └── 简单任务用 Haiku/GPT-5 mini/Gemini Flash，别用大炮打蚊子

2. Prompt Caching
   └── 缓存命中 = 50-90% 成本节省，Agent 架构必须考虑

3. Batch API
   └── 非实时任务用 Batch，50% 折扣

4. 精简 Prompt
   └── 减少不必要的上下文，每个 token 都是钱

5. 推理预算控制
   └── 不是每个问题都需要 reasoning_effort="high"

6. 输出长度控制
   └── 设置合理的 max_tokens，避免模型啰嗦
```

## 12. 2025-2026 关键趋势

### 趋势一：模型商品化，工程是护城河

模型能力趋同，差异化来自工程实现。以 Claude Code 为例：

```text
Claude Code 的成功 = 1.6% 模型能力 + 98.4% 工程实现

工程层面的关键：
- Prompt 工程：精心设计的系统提示和工具定义
- 缓存策略：利用 Prompt Caching 将成本降低 90%
- 上下文管理：智能选择哪些文件放入上下文
- 错误恢复：工具调用失败后的重试和降级策略
- 用户体验：流式输出、进度反馈、中断恢复
```

### 趋势二：开源模型缩小差距

2024 年初开源模型落后闭源 1-2 代，到 2026 年初 Llama 4、DeepSeek V3.2、Qwen 3.5 在多项基准上已与闭源持平。企业有了真正可行的私有部署选择，API 价格因竞争持续下降，微调和定制化成为差异化手段。

### 趋势三：上下文窗口爆炸式增长

从 2023 年的 4K 到 2026 年的 10M（Llama 4 Scout），上下文窗口增长了 2500 倍。"把整个代码库塞进上下文"成为可能，RAG 的角色从"必需"变为"优化手段"。但有效上下文利用仍是挑战——Lost in the Middle 现象依然存在。

### 趋势四：Prompt Caching 改变成本经济学

没有缓存时，10 轮 Agent 工具调用 × 100K 上下文 = 1M tokens 输入成本。有缓存后，100K 缓存命中（10% 价格）+ 10K 新增 ≈ 等效 20K tokens 成本，降低 98%。这就是 Claude Code 能在保持低成本的同时使用超大上下文的原因。

### 趋势五：推理模型成为新范式

传统范式是"更大的模型 → 更好的结果"（scaling law），新范式是"更多的推理时间 → 更好的结果"（inference-time compute）。o1 → o3 → o4-mini 是 OpenAI 的推理模型线，DeepSeek R1 证明推理能力可以通过 RL 训练获得，Claude Extended Thinking 将推理作为可控参数。未来方向是模型自动判断何时需要深度思考、何时快速回答。

### 趋势六：Agent 原生模型特性

2024 年模型支持 Function Calling（被动工具调用），2025 年支持 Computer Use（主动操作计算机），2026 年模型原生支持多步工具编排、自主错误恢复、长时间任务管理和 MCP 协议集成。模型正在从"被调用的工具"演变为"自主行动的 Agent 内核"。
