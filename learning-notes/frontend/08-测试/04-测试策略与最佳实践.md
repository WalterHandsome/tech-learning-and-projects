## 1. 测试金字塔

```
        /  E2E  \        ← 少量，验证关键用户流程
       / 集成测试 \       ← 适量，验证组件交互
      /  单元测试  \      ← 大量，验证独立逻辑
```

- 单元测试（70%）：工具函数、Hooks、Store、纯逻辑
- 集成测试（20%）：组件渲染、用户交互、API 集成
- E2E 测试（10%）：关键业务流程（登录、下单、支付）

## 2. 测试原则

```javascript
// ✅ 测试行为，不测试实现
// 好：测试用户看到什么、能做什么
it('shows error when form is invalid', async () => {
  render(<LoginForm />);
  await userEvent.click(screen.getByRole('button', { name: '登录' }));
  expect(screen.getByText('请输入用户名')).toBeVisible();
});

// ❌ 不好：测试内部状态
it('sets isError to true', () => {
  const { result } = renderHook(() => useForm());
  // 不要直接测试内部状态
});

// ✅ 每个测试独立，不依赖执行顺序
// ✅ 使用有意义的测试描述
// ✅ 遵循 AAA 模式：Arrange → Act → Assert
// ✅ Mock 外部依赖，不 Mock 被测代码
```

## 3. Mock 策略

```javascript
// 应该 Mock 的：
// - 网络请求（API 调用）
// - 定时器（setTimeout/setInterval）
// - 随机数/日期
// - 第三方服务

// 不应该 Mock 的：
// - 被测试的代码本身
// - 简单的工具函数
// - 数据转换逻辑

// MSW（Mock Service Worker）— 推荐的 API Mock 方案
import { setupServer } from 'msw/node';
import { http, HttpResponse } from 'msw';

const server = setupServer(
  http.get('/api/users', () => {
    return HttpResponse.json([{ id: 1, name: '张三' }]);
  }),
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

## 4. 覆盖率策略

```json
// vitest.config.ts 或 jest.config.js
{
  "coverage": {
    "provider": "v8",
    "reporter": ["text", "html", "lcov"],
    "thresholds": {
      "branches": 80,
      "functions": 80,
      "lines": 80,
      "statements": 80
    }
  }
}
```

- 追求有意义的覆盖率，而非 100%
- 关键业务逻辑覆盖率应更高
- 不要为了覆盖率写无意义的测试
