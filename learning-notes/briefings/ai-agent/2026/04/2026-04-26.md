### 1. Google Cloud Next 2026 全面复盘：Vertex AI 更名 Gemini Enterprise Agent Platform，Agent 平台三国杀格局成型

- **来源**：[SiliconANGLE](https://siliconangle.com/2026/04/25/googles-ai-agent-platform-takes-pole-position-work-remains/) / [Google Cloud Blog](https://cloud.google.com/blog/products/ai-machine-learning/introducing-gemini-enterprise-agent-platform) / [The Next Web](https://thenextweb.com/news/google-cloud-next-ai-agents-agentic-era) / [Computerworld](https://www.computerworld.com/article/4161990/gemini-enterprise-update-brings-ai-agents-into-collaborative-workflows.html)
- **标签**：`Google` `Gemini Enterprise` `Agent Platform` `Cloud Next`
- **评分**：时效 5 / 一手 5 / 相关 5 / 实用 4 = 总分 19
- **摘要**：Google Cloud Next 2026（4/22-25）本周落幕，核心动作是将 Vertex AI 更名为 Gemini Enterprise Agent Platform，同时将 Agentspace 吸收进统一的 Gemini Enterprise 产品。新平台整合了 Agent Designer（无代码构建）、Agent Inbox（管理 Agent 活动）、长时间运行 Agent、Skills、Projects 等能力。同时发布 TPU v8（8t/8i）、Knowledge Catalog 数据基础层，以及与 Wiz 合作的安全 Agent。SiliconANGLE 分析认为 Google 是唯一同时拥有前沿模型、差异化数据栈和 Agent 平台的超大规模云厂商。
- **影响**：至此，三大 Agent 平台格局正式成型：Google Gemini Enterprise Agent Platform（全栈整合，从芯片到应用）、OpenAI Codex + Workspace Agents（产品驱动，ChatGPT 生态）、Anthropic Claude Cowork + Managed Agents（开发者驱动，API 优先）。Google 的差异化在于数据层——Knowledge Catalog 将 BigQuery + Spanner 的元数据层与 Agent 平台打通，使 Agent 可以直接访问企业数据资产。这是 OpenAI 和 Anthropic 目前缺失的能力。
- **行动建议**：(1) 如果你的项目使用 GCP，评估从 Vertex AI 到 Gemini Enterprise Agent Platform 的迁移路径；(2) 关注 Agent Designer 的无代码构建能力是否适合快速原型；(3) 对比三大平台的 Agent 治理能力（IAM、审计、成本控制）。

### 2. Browser Harness：browser-use 团队发布极简 Agent 浏览器控制层，592 行 Python 给 LLM 完全自由

- **来源**：[GitHub](https://github.com/browser-use/browser-harness) / [Hacker News](https://news.ycombinator.com/item?id=47891260) / [pyshine](https://pyshine.com/browser-harness-ai-agent-browser-control/)
- **标签**：`开源` `Browser Agent` `CDP` `自修复`
- **评分**：时效 5 / 一手 4 / 相关 5 / 实用 5 = 总分 19
- **摘要**：browser-use 团队（78k+ Stars 的浏览器自动化库）发布 Browser Harness，采用与主库截然不同的设计哲学：不是用确定性启发式包装浏览器，而是给 LLM 最大自由度。整个项目仅 592 行 Python，直接通过 CDP WebSocket 连接 Chrome。核心创新是"自修复"机制——当 Agent 需要某个不存在的工具函数时（如 `upload_file()`），它会自己编写并注入到 `helpers.py` 中，运行时动态扩展能力。支持 domain-skills 目录，Agent 完成任务后自动生成可复用的技能文件。
- **影响**：这代表了 Browser Agent 设计的两个极端：browser-use 主库（数万行确定性代码）vs Browser Harness（592 行 + LLM 自由度）。Harness 的"Agent 自己写工具"模式与 04-23 简报中 Hermes Agent 的"自进化"理念一脉相承——让 Agent 在运行时扩展自身能力，而不是预先定义所有工具。对于 Claude Code 和 Codex 用户，这提供了一个极轻量的浏览器控制方案。
- **行动建议**：(1) 如果你的 Agent 需要浏览器操作，评估 Browser Harness 的极简方案 vs browser-use 主库的全功能方案；(2) 关注 domain-skills 目录中的社区贡献技能。

### 3. Mark Nottingham 深度博文：Agentic 叙事中缺失的关键角色——用户代理

- **来源**：[mnot.net](https://www.mnot.net/blog/2026/04/24/agents_as_collective_bargains) / [Hacker News](https://news.ycombinator.com/item?id=47902339)（58 分 / 60 评论）
- **标签**：`Agentic Web` `用户代理` `信任模型` `行业思考`
- **评分**：时效 5 / 一手 5 / 相关 4 / 实用 3 = 总分 17
- **摘要**：HTTP 规范的核心贡献者 Mark Nottingham 发表深度博文，指出当前 Agentic AI 叙事中缺失了一个关键角色：真正代表用户利益的"用户代理"（User Agent）。他从工具信任的历史出发——螺丝刀不会自己决定做别的事——论证现代互联网设备已经打破了这种信任假设。电视监控用户、Meta 解密用户流量、Microsoft 改变 Outlook 模型——用户以为"我的设备为我工作"，实际上设备同时服务于多方利益。在 AI Agent 时代，这个问题将被极度放大：Agent 代表谁的利益？用户的、厂商的、还是第三方的？
- **影响**：这篇文章触及了 Agent 生态的根本性问题。当 OpenAI Workspace Agents、Google Gemini Enterprise、Anthropic Managed Agents 都在推"企业 Agent"时，谁来确保 Agent 真正代表终端用户的利益？Nottingham 的分析暗示需要一个新的"用户代理"层——类似于浏览器在 Web 中扮演的角色，但在 Agent 时代需要重新定义。这对 Agent 框架设计者和政策制定者都有深远影响。
- **行动建议**：阅读原文，思考你的 Agent 产品中用户利益与平台利益的边界。

## 📄 值得关注的论文 / 基准

### LamBench：Lambda 演算基准测试挑战 LLM 推理极限

- **作者/机构**：Victor Taelin（HVM/Bend 创始人）
- **摘要**：LamBench 使用 Lambda 演算作为基准测试 LLM 的推理和推断能力。Lambda 演算是图灵完备的最小形式系统，测试 LLM 能否执行纯粹的符号计算——这是当前 LLM 最薄弱的环节之一。HN 获得 135 分 / 40 评论。
- **关键贡献**：提供了一个极简但深刻的推理基准，与传统 Coding 基准（SWE-bench、HumanEval）互补——后者测试实用编程能力，LamBench 测试纯粹的计算推理能力。
- **链接**：[LamBench](https://victortaelin.github.io/lambench/) / [GitHub](https://github.com/VictorTaelin/LamBench)

## 💬 社区讨论

### Agent 记忆层的开源爆发：Stash、Wuphf、MemPalace 同期涌现

本周 HN 上同时出现多个 Agent 记忆层开源项目：Stash（165 分，通用持久记忆）、Wuphf（229 分，Karpathy 风格 Markdown+Git Wiki）。这反映了社区对 Agent 记忆基础设施的强烈需求——当前主流 Agent（Claude Code、Codex、Cursor）的记忆能力仍然有限，每次会话基本从零开始。Karpathy 此前提出的"LLM 原生知识基底"概念正在被多个独立团队实现。与 04-23 简报中 Hermes Agent 的"持久记忆 + 自动技能生成"形成呼应。

### HATS：用结构化辩论对抗 LLM 的"同意偏见"

HATS 项目在 HN 上引发了关于多 Agent 协作模式的讨论。核心洞察：单个 LLM 倾向于同意用户、过度自信、忽略盲点。HATS 通过六顶思考帽框架引入"结构化冲突"——六个 Agent 分别扮演事实、直觉、风险、机会、创意、综合角色，强制从多角度审视问题。支持 OpenAI/Claude/Gemini/Ollama 多模型混合，每个 Agent 可选不同模型。
