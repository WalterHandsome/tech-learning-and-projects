# Karpathy 编码行为准则

> 来源: [forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)
> 适配: Walter Wang 的技术笔记和 demo 项目

## 适用场景

- 编写或修改 demo 项目代码（crewai-multi-agent-demo、langgraph-mcp-agent-demo 等）
- 编写脚本（scripts/ 目录）
- 审查或重构现有代码
- 任何涉及代码生成的任务

**不适用**：纯文档编辑、简报采集、配置文件修改。这些场景由 steering 规则覆盖。

## 四大原则

### 1. 先想再写

不要假设，不要隐藏困惑，暴露 tradeoff。

- 明确说出你的假设。不确定就问
- 多种理解时列出方案，不要默默选一个
- 有更简单的方案就提出来，即使用户没问
- 搞不清楚就停下来，说出哪里不清楚

### 2. 简洁优先

只写解决当前问题的最少代码。

- 不加没要求的功能、抽象、"灵活性"
- 单次使用的逻辑不提前抽象
- 200 行能用 50 行写就重写
- 不为不可能的场景加错误处理
- 自检：一个资深工程师会说这过度设计了吗？如果会，简化

### 3. 精准修改

只动必须动的代码，只清理自己制造的垃圾。

编辑现有代码时：
- 不"顺手改进"旁边的代码、注释、格式
- 不重构没坏的东西
- 匹配现有风格，即使你觉得不好
- 发现无关死代码，提一句但不删

你的改动产生的孤儿代码（import、变量、函数）要清理。别人的不要动。

**检验标准**：每一行改动都能直接追溯到用户的请求。

### 4. 目标驱动

把任务转化为可验证的目标，而不是模糊的步骤。

- "修 bug" → 先写复现测试，再修，确认测试通过
- "加功能" → 先定义成功标准，再实现，逐步验证
- "重构 X" → 确保重构前后测试都通过

多步任务列计划：

```
1. [步骤] → 验证：[怎么确认成功]
2. [步骤] → 验证：[怎么确认成功]
```

强成功标准让你能独立循环。弱标准（"让它能用"）需要反复确认。

## 本项目特殊规则

- demo 项目的目的是**演示**，保持代码简洁直观
- 学习笔记中的代码示例追求**清晰易懂**，不追求面面俱到
- scripts/ 下的工具脚本追求**可靠和幂等**，加必要的错误处理

## 验证

应用本准则后自检：
- [ ] 每行改动都能追溯到用户请求
- [ ] 没有添加未要求的抽象或功能
- [ ] 假设已明确说出
- [ ] 有可验证的成功标准
